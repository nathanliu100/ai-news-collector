## 一、app.js 715 行逐段审视——我找到了 7 个隐藏问题

> 这是 20 天以来第一次认真审读前端代码而不只是"看一眼行数"。

### 1a. 🔔 传闻标签（`rumor`）在整个 JS 中不存在

**问题定位**：`structureNewsBlocks()` 函数（L261-298）负责把 H3 标题按 emoji 分类为 `high`/`medium`/`low`/`unknown`。

```javascript
if (text.indexOf('\uD83D\uDD34') >= 0) level = 'high';     // 🔴
else if (text.indexOf('\uD83D\uDFE1') >= 0) level = 'medium'; // 🟡
else if (text.indexOf('\uD83D\uDFE2') >= 0) level = 'low';    // 🟢
// ❌ 没有 🔔 的处理！
```

今天的 4/27 日报有 **4 条 🔔 传闻**（Google I/O、Anthropic 估值、Cursor 套壳、Altman 辞去 Helion）。它们全部被标记为 `data-level="unknown"`。

**影响**：
- Filter 按钮（Critical / Notable / FYI）点击后，🔔 传闻新闻永远**不会被显示**——因为 `unknown` 不匹配任何 filter level
- 只有 "All" 模式才能看到它们
- 用户不知道这些新闻"消失了"，会以为网站 bug

**修复方案**：加一个 `else if` 处理 🔔 → `level = 'rumor'`，同时在 filter 按钮中新增一个 "📡" 按钮（或者让 🔔 归入 `low` 也行，看产品决策）。

**工时**：10 分钟。**这个问题已经存在了至少 10 天（从 4/15 建立信源分级体系起）。**

### 1b. Stats 统计遗漏 🔔

**问题定位**：`updateStats()` 函数（L475-491）统计 `🔴`/`🟡`/`🟢` 的数量。

```javascript
var highs = (md.match(/^### 🔴/gm) || []).length;
var meds = (md.match(/^### 🟡/gm) || []).length;
var lows = (md.match(/^### 🟢/gm) || []).length;
```

🔔 完全不在统计范围内。今天 22 条新闻的 stats ribbon 会显示 `共 22 | 🔴 4 | 🟡 11 | 🟢 3`——加起来是 18，剩下 4 条 🔔 在统计上"蒸发了"。

**用户感知**："为什么共 22 条但 4+11+3=18？少了 4 条？"

**修复方案**：stats ribbon 加一个 🔔 计数，或者在 `Total` 旁边显示 `(含 4 条传闻)`。

### 1c. X/KOL 统计使用了错误的 emoji

**问题定位**：`updateStats()` 的 X 平台统计：

```javascript
var xkol = (md.match(/🐦/g) || []).length;
```

但实际新闻 Markdown 中的 X 平台标记不是 🐦——查看今天的日报，来源标注格式是 `[🏢 官方]` / `[📰 媒体]`，X 的标记是 `X/KOL`（纯文本）或使用 `𝕏` 符号。**🐦 这个 emoji 在任何日报中从未出现过。**

**结果**：Stats ribbon 中的 𝕏 数字永远是 0。

**实际来源格式**（从 4/27 日报）：
- `[🏢 官方]` → 7 条
- `[📰 媒体]` → 12 条
- 直接写 `X/KOL` → 3 条

**修复方案**：改为匹配 `X\/KOL` 或新闻正文中实际使用的标记。需要先确认 COLLECTOR_PROMPT 的来源标注规范。

### 1d. `scanFiles()` 对周日没有 fallback

**问题定位**：History Panel 的文件扫描（L582-615）：

```javascript
var isWe = (d.getDay() === 0 || d.getDay() === 6);
var primary = isWe ? ('news/' + ds + '-weekend.md') : ('news/' + ds + '.md');
var r = await fetch(primary, { method: 'HEAD' });
if (r.ok) {
    files.push({ date: ds, weekend: isWe });
    continue;
}
if (isWe) {
    // weekend 版不存在就 fallback 到普通版
    var r2 = await fetch('news/' + ds + '.md', { method: 'HEAD' });
    if (r2.ok) files.push({ date: ds, weekend: false });
}
```

周日（`getDay() === 0`）被标记为 `isWe = true`，所以先查 `-weekend.md`，再查普通 `.md`。但实际上：
- 周六发 `-weekend.md` ✅
- 周日什么都不发 ❌

所以周日在 History Panel 中就是一个**空洞**——没有条目。这不是 bug（确实没有文件），但体验上看到日期列表跳过周日会让人困惑。

**和 Day 19 提出的"周日回退"是同一个问题的两个面**——loadNews 层面和 scanFiles 层面都需要处理。

### 1e. CREW 对象缺少向后兼容

已知问题（P3，Day 17 提出）。CREW 对象没有 `trump` 键，4/17-4/23 的旧新闻 crew 评论渲染为无头像无名字的匿名评论。

**注意到的新细节**：`injectCrewComments()` L456 的 fallback 行为是 `CREW[c.key] || { name: c.key, emoji: '💬', avatar: '' }`——所以旧 Trump 评论会显示 `name: "trump"`（小写原始 key），emoji 是通用的 💬，头像为空字符串。

**不是灾难但不优雅。** `name: 'trump'` 看起来像 bug。

### 1f. 30 天扫描上限

`scanFiles()` 只扫描最近 30 天。网站已经运行 28 天（3/30 → 4/27）。**再过 2 天，最早的日报（3/30）就会从 History Panel 中消失。**

这不是今天要修的事，但需要意识到：网站的历史内容正在"自然过期"。如果有用户（比如新来的同事）想看 Week 1 的日报，30 天后就找不到了。

**两个方向**：
1. 改为扫描全部文件（但 HEAD 请求会很多）
2. 生成一个静态的 `index.json` 文件列出所有日期（一劳永逸）

**方向 2 更好**——可以在每天 automation 结束后自动更新 `news/index.json`。这样 scanFiles 只需要一次 fetch。

### 1g. `marked.js` 加载时机的潜在竞态

```html
<script defer src="https://cdn.jsdelivr.net/npm/marked/marked.min.js"></script>
<!-- ... -->
<script defer src="assets/app.js"></script>
```

两个 `defer` 脚本按文档顺序执行，所以理论上 `marked` 应该在 `app.js` 之前加载完成。但如果 CDN 延迟或失败（国内访问 jsdelivr 不稳定），`marked.parse()` 会抛 `undefined is not a function`。

**目前没有 fallback**——如果 marked.js 加载失败，用户看到的是一个永久 loading 状态。

**低优先级但值得记录。**

## 六、竞品视角：Hacker News 的"反设计"哲学

### 6a. 为什么 HN 不做推送

Hacker News 没有推送通知、没有企微群、没有邮件订阅（有第三方做的，但官方没有）。它只有一个 30 年不变的橙色列表页面。

**但 HN 依然有大量日活。为什么？**

因为 HN 的用户是**主动型消费者**——他们每天固定时间打开 HN，是他们日常工作流的一部分。这和 Morning Brew 的"推送型"模式完全相反。

### 6b. AI News Collector 的用户是哪种？

Day 19 诊断过："80% 的用户价值在企微推送端完成"。这意味着 Nate 的同事是**被动型消费者**——他们不会主动打开网站，需要被推送"投喂"。

**但有一个例外——Nate 自己。** Nate 是唯一的"主动型"用户——他每天打开网站检查日报质量，他是内容创作者而不是消费者。

**产品启示**：网站的设计应该服务两类截然不同的用户：
1. **Nate（创作者）**：需要看到全部细节——stats、filter、Crew、锚点、History
2. **同事（消费者）**：只需要"今天最重要的 3-5 件事"

**当前网站对两类用户用同一个 UI——这可能是"信息过载"感的根本原因。**

一个轻量的想法：**首屏优先展示"今日必看"板块，其余内容折叠**。这样消费者进来就看到最关键的信息，想深入再展开。类似 Apple News 的"Top Stories"优先展示逻辑。

**优先级**：P3 产品方向。不急，但值得在下一次前端改动时考虑。

## 九、大胆想法

**"news/index.json"——一个文件解决三个问题**

当前的 `scanFiles()` 用 30 次 HEAD 请求来发现历史文件。这有三个问题：
1. 30 天上限（再过 2 天最早的日报消失）
2. 每次打开 History Panel 都要等扫描（慢）
3. 国内网络环境下 30 次 fetch 可能有超时

**解法**：每天 automation 结束后，自动生成一个 `news/index.json`：

```json
{
  "files": [
    { "date": "2026-04-27", "type": "daily", "stories": 22, "critical": 4 },
    { "date": "2026-04-25", "type": "weekend", "stories": 10, "critical": 0 },
    { "date": "2026-04-24", "type": "daily", "stories": 22, "critical": 4 },
    ...
    { "date": "2026-03-30", "type": "daily", "stories": 8, "critical": 1 }
  ],
  "weeklies": ["weekly-2026-W17.md", "weekly-2026-W16.md", "weekly-2026-W14.md"],
  "updated": "2026-04-27T09:15:00+08:00"
}
```

**一次 fetch 替代 30 次 HEAD**。History Panel 秒开，无上限，还能显示每天的新闻数量和 🔴 数量——让历史列表本身就有信息量。

**额外收益**：这个 JSON 可以被企微推送脚本读取，可以被周报 automation 读取——它是整个系统的"元数据层"。

**优先级**：P2。15 分钟可以搞定（生成脚本 + scanFiles 改写）。
