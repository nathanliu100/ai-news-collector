### 1. 周末空窗期：网站在非工作日"死"了吗？

这对不同用户的影响：

- 但——实际用户周末访问的概率本来就极低（这是内部工具，不是 C 端产品）
- **结论：目前不是问题。但如果未来想扩大受众，周末内容策略需要考虑**

**如果要解决（V2 阶段的可选方案）**：

- **方案 C：不管它**——诚实地说，周末空窗对当前用户群体影响几乎为零

### 2. 方案细化：新闻锚点系统的完整设计

> 这是 P0 优先级第一项，被提了 4 天了。今天不再说"应该做"，而是把**怎么做**写清楚，精确到代码层面，下周一直接执行。

#### 2a. 锚点 ID 生成规则

当前新闻的 Markdown 结构：
```markdown
### 🔴 [Anthropic 发布 Claude Mythos Preview，限 50 家企业使用](https://...)
**来源**：Anthropic 官方 [🏢 官方] | **时间**：2026-04-07
正文段落...
```

`marked.js` 解析后生成的 HTML：
```html
<h3>🔴 <a href="...">Anthropic 发布 Claude Mythos Preview，限 50 家企业使用</a></h3>
```

**ID 生成策略**：
- 在 `marked.js` 的 renderer 里 override `heading` 方法
- 对 H3 标题提取纯文本（去掉 emoji 和链接）
- 使用 `encodeURIComponent()` 或简单的 slugify（中文直接用 `encodeURIComponent`，英文用 kebab-case）
- 格式：`#news-{index}`（简单序号）或 `#news-{slug}`（语义化）

**推荐用序号方案**：`#news-1`, `#news-2`, ..., `#news-22`
- 原因：中文 slug 太长且 URL 不美观；序号简洁，和「今日必看」的编号对应
- 缺点：如果新闻顺序变化（比如重新排序），锚点会失效。但实际上日报生成后内容不会变，所以没问题

#### 2b. 锚点交互设计

```
┌─────────────────────────────────────────────┐
│ 🔴 Anthropic 发布 Claude Mythos...    🔗    │
│ ────────────────────────────────────────     │
│ 来源：Anthropic 官方 | 时间：2026-04-07     │
│ 正文正文正文...                              │
└─────────────────────────────────────────────┘
```

- 🔗 图标默认不可见（或 opacity: 0.3）
- **桌面端**：hover 新闻条目时，🔗 图标淡入（opacity: 1），位于 H3 标题行右侧
- **移动端**：🔗 图标始终可见但小一点（12px），位于标题行末尾
- 点击 🔗：
  1. 复制 `https://nathanliu100.github.io/ai-news-collector/?date=2026-04-10#news-3` 到剪贴板
  2. 显示一个轻量 toast："链接已复制 ✓"（2秒后消失）
  3. 同时 URL 栏更新（`history.replaceState`）

#### 2c. 页面加载时的锚点处理

```javascript
// 在 Markdown 渲染完成后
if (window.location.hash) {
    const target = document.querySelector(window.location.hash);
    if (target) {
        // 延迟 300ms 等渲染完成
        setTimeout(() => {
            target.scrollIntoView({ behavior: 'smooth', block: 'start' });
            // 高亮动画：目标新闻条目闪一下
            target.closest('.news-item')?.classList.add('highlight-flash');
            setTimeout(() => {
                target.closest('.news-item')?.classList.remove('highlight-flash');
            }, 2000);
        }, 300);
    }
}
```

高亮 CSS：
```css
.highlight-flash {
    animation: flash-highlight 2s ease-out;
}

@keyframes flash-highlight {
    0% { background: var(--accent-soft); }
    100% { background: transparent; }
}
```

### 3. 方案细化：OG Meta 标签的实现策略

#### 3a. 核心困难

GitHub Pages 是纯静态托管，不支持 SSR。这意味着：
- 不能根据当天日期动态生成 `<meta property="og:description">`
- 微信/企微分享预览抓取的是 HTML 原文中的固定 meta 标签
- **动态 OG 在纯前端方案中是不可能的**

#### 3b. 分级策略

**Level 1：静态 OG（今天就能做）**

```html
<meta property="og:type" content="website">
<meta property="og:title" content="AI 新闻速递 | AI Daily Briefing">
<meta property="og:description" content="每日 AI 前沿新闻，覆盖模型发布、AI+设计、AI+游戏、市场动态、工具产品等 8 大领域。">
<meta property="og:image" content="https://nathanliu100.github.io/ai-news-collector/og-image.png">
<meta property="og:url" content="https://nathanliu100.github.io/ai-news-collector/">
<meta name="twitter:card" content="summary_large_image">
```

需要额外做一张 `og-image.png`（1200×630px），品牌感的：
- 背景：深色调 + 编辑风 texture
- 文字：「AI 新闻速递」大标题 + 「每日更新 · 8 大领域 · 双语版本」副标题
- 可以用 image_gen 一次性生成，不需要每天更新

**这已经解决了 80% 的问题**——从"分享时什么都没有"变成"有品牌标题 + 描述 + 图片"。

**Level 2：每日动态 OG（GitHub Actions 方案）**

在每天的新闻生成 workflow 里，加一步：
1. 从当天 Markdown 中提取「今日必看」的前 3 条标题
2. 用 `sed/awk` 直接替换 `index.html` 里的 `<meta>` 标签
3. 生成当日的 OG 图片（可以用 Puppeteer 截图或简单的 canvas 脚本）
4. 把更新后的 `index.html` 和 OG 图片一起 push

```yaml
# 在 GitHub Actions workflow 里加一步
- name: Update OG meta
  run: |
    TODAY=$(date +%Y-%m-%d)
    HEADLINE=$(head -20 news/$TODAY.md | grep "^\*\*" | head -1 | sed 's/\*//g')
    sed -i "s|<meta property=\"og:description\" content=\".*\"|<meta property=\"og:description\" content=\"$TODAY: $HEADLINE\"|" index.html
```

**风险**：sed 替换 HTML 容易出错，而且微信的 OG 缓存很激进（可能 24 小时才更新）。

**建议**：先做 Level 1，验证分享体验有提升后再考虑 Level 2。

**预估工作量**：Level 1 = 30 分钟（加 meta 标签 + 做一张 OG 图），Level 2 = 1-2 小时

### 4. 阅读体验显微镜：逐像素审视排版

> 前几天一直在想"功能"，今天退一步，用设计师的眼睛看**排版细节**——那些用户说不出来但"感觉不对"的地方。

打开 `index.html` 的 CSS，逐项审视：

#### 4a. 行间距和段间距的节奏感

当前设置：
```css
body { line-height: 1.7; font-size: 15px; }
.news-content h3 { margin: 24px 0 6px; }
.news-content p { margin: 4px 0 20px; }
```

**问题**：H3 和紧跟其后的 `<p>`（来源 + 正文）之间只有 6+4=10px，而段落底部有 20px margin。这导致**新闻标题和正文之间太紧，正文和下一条之间太松**——视觉节奏是"紧-松-紧-松"，而不是更自然的"松-紧-紧-松"分组。

**优化建议**：
```css
.news-content h3 { margin: 32px 0 8px; }  /* 上方加大，标题前"呼吸"更多 */
.news-content p { margin: 4px 0 12px; }    /* 段落间距收紧，让标题+正文更紧凑成组 */
```

核心原则是**格式塔的邻近法则（Law of Proximity）**：属于同一条新闻的元素应该靠得更近，不同新闻之间的间距应该更大。

#### 4b. 新闻来源行的视觉层级

当前：
```markdown
**来源**：Anthropic 官方 [🏢 官方] | **时间**：2026-04-07
```

渲染后是一行加粗文本，和正文段落在视觉上几乎一样。

**问题**：来源行是**元数据**，不是内容。它的视觉权重应该比正文低。

**优化方向**：
```css
/* 来源行：用更小的字号、更浅的颜色、monospace 字体 */
.news-content h3 + p:first-of-type {
    font-size: 12px;
    color: var(--text-tertiary);
    font-family: 'JetBrains Mono', 'Inter', monospace;
    letter-spacing: 0.3px;
    border-bottom: 1px solid var(--border);
    padding-bottom: 8px;
    margin-bottom: 8px;
}
```

这个选择器有局限性（依赖 DOM 结构），更精确的做法是在 `marked.js` 的 renderer 里对"来源"行加一个 class。

但——**能不能用纯 CSS 匹配？**

如果来源行的 HTML 渲染后总是 `<p><strong>来源</strong>：...` 的格式，可以用：
```css
.news-content p:has(> strong:first-child) {
    /* 可能匹配到太多段落 */
}
```

不太靠谱。还是在 JS 渲染后做一次 DOM walk 更稳：
```javascript
document.querySelectorAll('.news-content h3 + p').forEach(p => {
    if (p.textContent.startsWith('来源') || p.textContent.startsWith('Source')) {
        p.classList.add('news-meta');
    }
});
```

**工作量**：30 分钟，可以和锚点功能一起做。

#### 4c. 「今日必看」有序列表的数字设计

当前渲染为标准的 `<ol><li>` 列表，数字是浏览器默认的样式。

**对比 Morning Brew 的头条列表**：它用的是大号数字 + 手动排版，数字本身成为视觉元素。

**一个简单的 CSS-only 升级**：
```css
.news-content > ol:first-of-type {
    counter-reset: must-read;
    list-style: none;
    padding-left: 0;
}

.news-content > ol:first-of-type > li {
    counter-increment: must-read;
    position: relative;
    padding-left: 48px;
    margin-bottom: 16px;
}

.news-content > ol:first-of-type > li::before {
    content: counter(must-read);
    position: absolute;
    left: 0;
    top: 2px;
    width: 32px;
    height: 32px;
    background: var(--accent);
    color: white;
    font-family: 'Playfair Display', serif;
    font-size: 16px;
    font-weight: 700;
    display: flex;
    align-items: center;
    justify-content: center;
    border-radius: 50%;
}
```

这让「今日必看」的数字从"浏览器默认的 1. 2. 3."变成**红色圆形编号**，一眼就知道这是今天的重头戏。

**风险**：CSS 选择器 `.news-content > ol:first-of-type` 可能会误匹配到其他列表。需要确认 Markdown 渲染后「今日必看」的 `<ol>` 确实是第一个（大概率是，因为它在「⚡今日必看」H2 之后紧跟着就是有序列表）。

**工作量**：15 分钟纯 CSS 改动。

#### 4d. 表格（数据快照）的移动端体验

当前：
```css
@media (max-width: 640px) {
    .news-content table {
        display: block;
        overflow-x: auto;
        white-space: nowrap;
    }
}
```

移动端表格变成横向滚动——**可以用，但不优雅**。数据快照通常只有 5-6 行 × 3 列，信息量不大。

**替代方案：移动端把表格转成卡片列表**

```css
@media (max-width: 640px) {
    .news-content table thead { display: none; }
    .news-content table tr {
        display: block;
        margin-bottom: 12px;
        padding: 12px;
        background: var(--surface-alt);
        border-radius: 8px;
    }
    .news-content table td {
        display: block;
        padding: 4px 0;
        border: none;
    }
    .news-content table td::before {
        content: attr(data-label);
        font-weight: 600;
        font-size: 11px;
        color: var(--text-tertiary);
    }
}
```

**但**——这需要给 `<td>` 加 `data-label` 属性，纯 Markdown → HTML 渲染做不到。需要在 JS 后处理中遍历表格，把 `<th>` 内容写到对应 `<td>` 的 `data-label` 上。

```javascript
document.querySelectorAll('.news-content table').forEach(table => {
    const headers = [...table.querySelectorAll('th')].map(th => th.textContent);
    table.querySelectorAll('td').forEach((td, i) => {
        td.setAttribute('data-label', headers[i % headers.length] + '：');
    });
});
```

**工作量**：30 分钟（JS + CSS），效果在移动端会明显提升。

## 💡 今天的大胆想法

**"周六编辑精选"自动推送**

既然周末没有新日报，何不利用这个时间窗口做点不一样的事？

设想一个周六上午自动跑的 automation：
1. 回顾本周 5 天的所有新闻（~100-110 条）
2. 从中选出「最值得周末慢读的 5 条」——标准不是"最重大"（那些工作日已经看过了），而是**"最值得深思"**：有深度分析的、有争议性的、有长期影响的
3. 每条配一段 50-100 字的"周末视角"评论——比工作日的新闻摘要更轻松、更有态度
4. 推送到企微群，标题是「☕ 周末慢读 | AI 本周值得再想想的 5 件事」

**这解决了两个问题**：
- 填补周末空窗期
- 提供和工作日完全不同的阅读体验（快速扫描 vs 慢读深思），让产品有"呼吸感"

**工作量**：一个新的 automation + 一段 prompt，不需要改前端。
