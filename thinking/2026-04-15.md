### 1. 信源可信度分级体系——首次实战的表现打分

> 这是 4/14 思考日志催生的最重要产出。今天是它在生产环境的第一天。怎么样？

#### 1a. 做得好的（值得庆祝）

**GPT-6 假新闻拦截是教科书级的成功案例。**

还原 4/13 → 4/14 → 4/15 的信息演化：

```
4/13：🔴 GPT-6 明日发布（"官方确认"）   ← 误报
4/14：🟡 GPT-6 预计 4 月发布 + ⚠️勘误    ← 手动纠正
4/15：🔔 GPT-6 仍在预训练/后训练阶段     
      + 信源声明："中文自媒体 AI 投毒假新闻，未收录"   
      + 编辑观察区再次强调 ← 系统性防线
```

从"被动纠错"（4/14）进化到"主动拦截 + 公开声明"（4/15），只用了 1 天。**信源体系把一次尴尬的误报变成了产品的信任资产。**

具体做得好的地方：
- ✅ 引言区的 `🛡️ 来源可信度` 统计一目了然：T1 14 条 / T2 4 条 / 📡 传闻 2 条 / T3 0 条
- ✅ 每条新闻都带了 `[🏢 官方] [🟩 T1]` 或 `[📰 媒体] [🟨 T2]` 标签
- ✅ 🔴 重大新闻附带"✅ 已验证"+ 具体交叉验证来源
- ✅ 传闻标注为 `🔔` 而非 `🟡`，**降低了读者将传闻误判为确认消息的风险**
- ✅ 编辑观察区设置了"⚠️ 信源警示"段落，专门点名 AI 投毒

#### 1b. 可以更好的（微调方向）

**问题 1：Tier 标签的视觉辨识度**

当前写法：
```
[🏢 官方] [🟩 T1]
```

在正文密集的新闻块里，`[🟩 T1]` 这个标签很容易被正文淹没——它和链接文字、emoji 标签混在一起，需要 2 秒才能定位到。

**对比 Scanner 的阅读节奏**：Scanner 扫 H3 标题 + 第一行关键词。如果 Tier 标签在第二行（来源行），Scanner 根本不会看到。

**微优化方向**：在 H3 标题末尾直接显示 Tier 等级（和 🔴🟡🟢 重要性标签同行），让两个关键判断维度"重要性 × 可信度"在同一行完成。

```markdown
# 当前：
### 🔴 [Anthropic Claude Mythos Preview：太危险而不敢公开的 AI]()
**来源**：Anthropic 官方 System Card [🏢 官方] [🟩 T1] | ...

# 优化后（概念）：
### 🔴🟩 [Anthropic Claude Mythos Preview：太危险而不敢公开的 AI]()
**来源**：Anthropic 官方 System Card | ...
```

但这个改动要小心——标题区域的 emoji 已经很密了（重要性 + 时效性 + 💎 + 🎯）。再加一个 Tier 标签可能会造成视觉噪音。**需要实际 A/B 看看，不急着做。P2 级别。**

**问题 2：传闻 `🔔` 标签在 stats ribbon 中没有体现**

4/15 日报有 2 条传闻（GPT-6 Spud + DeepSeek V4），但 stats ribbon 只统计 🔴🟡🟢 三个等级，`🔔` 被遗漏。前端 `updateStats()` 函数没有对 `🔔` 做 regex 匹配。

```javascript
// 当前：
var highs = (md.match(/^### 🔴/gm) || []).length;
var meds = (md.match(/^### 🟡/gm) || []).length;
var lows = (md.match(/^### 🟢/gm) || []).length;
// 缺少：var rumors = (md.match(/^### 🔔/gm) || []).length;
```

**影响**：`totalCount` 目前靠匹配所有 `### ` 来计数（包含 🔔），但分项不加 🔔，导致 `高 + 中 + 低 ≠ 总计`。用户可能不会注意到，但数据完整性有瑕疵。

**修复量**：~15min。在 stats ribbon 里新增一个 `🔔` 指示器（蓝色），JS 里加一行 regex。**P1 级别**——因为这是数据展示正确性的问题。

**问题 3：传闻新闻的 filter 归属**

`setFilter()` 目前把新闻分为 `high`/`medium`/`low` 三个 level。`🔔` 新闻在 `structureNewsBlocks()` 里会被归为 `unknown`（因为 text 里没有 🔴🟡🟢），filter 任何一个等级都不会显示它。

```javascript
// structureNewsBlocks() 里：
if (text.indexOf('🔴') >= 0) level = 'high';
else if (text.indexOf('🟡') >= 0) level = 'medium';
else if (text.indexOf('🟢') >= 0) level = 'low';
// 🔔 未处理 → level = 'unknown' → 只在 "All" 下可见
```

**用户体验问题**：当用户点击 "Critical" filter 时，2 条传闻（GPT-6、DeepSeek V4）消失了。但传闻新闻可能恰恰是用户最想跟踪的——**"哪些大事快来了"**。

**解决方案选项**：
- A：把 🔔 归入 `medium`（和 🟡 同级）→ 简单，但模糊了"传闻"和"确认中等重要"的区别
- B：新增第四个 filter level `rumor` → 干净，但增加了 UI 复杂度（桌面 5 个 pill，移动端 5 个 m-pill）
- C：把 🔔 归入 `medium` 但在视觉上做区分（比如虚线边框）→ 折中方案

**建议选 A 作为短期方案（5min），长期看 B 更对。** P1 级别。

- 如果要修 `🔔` filter bug，需要在 1997 行里找到 `structureNewsBlocks()`（第 1618 行）和 `updateStats()`（第 1764 行）和 `applyFilter()`（第 1799 行）——三个不同位置

**不需要引入构建工具**（那是过度工程化）。只需要：
```

app.js       → 所有 JS（~610 行）
```

**工时**：30min 的 copy-paste + 验证。低风险高回报。

### 4. 信源体系对用户心理的影响——从"推送可信度"视角

> 这个维度之前完全没有被触及。信源体系改善的不只是网站内容质量，更关键的是**企微推送的信任链**。

#### 4a. 推送场景下的信任链

同事在企微群收到推送时，心理决策路径是：

```
看到标题 → "这可信吗？" → 如果信 → 转发/汇报
                         → 如果不信 → 忽略

4/13 之前（无信源标记）：
  "GPT-6 明天发布" → 信（标题写了嘛） → 转发 → 第二天被打脸 → 信任 -50

4/15 之后（有信源标记 + 传闻标签）：
  "🔔 GPT-6 仍在预训练阶段 [📡 传闻]" → 知道是传闻 → 不轻易转发 → 信任 +10
```

**信源体系最大的价值不在网站上（用户已经点进来了），而在推送里（用户还没点进来就要做"信不信"的判断）。**

#### 4b. 企微推送需要同步升级

如果推送里只显示标题，不显示 Tier 标签，那信源体系的价值打了五折。

建议推送格式升级：
```
⚡ AI 新闻速递 2026-04-15（🟩×14 🟨×4 📡×2）

🔴✅ Anthropic 发布 Claude Mythos Preview 但拒绝公开
🔴✅ Project Glasswing：11 巨头组建 AI 网安联盟
🔴✅ 斯坦福 HAI 2026 AI Index Report
🔴✅ Sam Altman 住宅 3 天内遭两次袭击

📡 GPT-6 / DeepSeek V4 均为未验证传闻

📰 完整日报(20条) → nathanliu100.github.io/ai-news-collector/
```

**变化**：
1. 标题行添加信源统计概览（🟩×14 …）
2. 每条必看后加 `✅`（已验证）或 `📡`（传闻）
3. 传闻新闻单独一行，明确标注"未验证"
4. 这让推送自带"这条消息靠谱吗"的信息——**同事不需要点进网站就能判断**

**工作量**：修改 `notify-wecom.sh`（或推送相关 prompt），约 20min。**P1 级别。**

```

本周采集：140 条
├── 🟩 Tier 1 权威：98 条（70%）
├── 🟨 Tier 2 参考：32 条（23%）
├── 📡 传闻：8 条（6%）
```

**为什么值得做**：
1. 量化"我们的新闻有多靠谱"——对 Nate 向老板汇报有价值（"我们的 AI 新闻平台 🔴 新闻交叉验证率 100%"）

3. 可以嵌入周报末尾，变成一个固定栏目

**落地路径**：
- V1：在周报 prompt 里加"统计本周信源分布"指令 → 0 前端代码
