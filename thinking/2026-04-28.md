## 一、灾难推演：如果明天出事了

### 1a. 为什么要做这个

PocketOS 的 9 秒删库事件不是别人的故事——它提醒我们审视自己的系统。AI News Collector 每天由 automation 自动运行、自动 git push、自动企微推送。**自动化程度越高，出错时的爆炸半径越大。**

让我逐一推演每种灾难场景：

### 1b. 场景 1：Automation 生成了有严重错误的日报

**触发条件**：AI 搜索结果出现幻觉、信源分级失效、写入了完全虚假的 🔴 新闻。

**当前防御层**：
- ❌ **零人工审核**——automation 9:00 跑完直接 push + 推送
- ✅ 信源分级体系（Tier 1/2/3）+ 交叉验证规则写在 COLLECTOR_PROMPT.md
- ✅ 反 AI 投毒规则（中文自媒体独有 + 英文圈静默 → 不收录）
- ❌ **没有"暂存"机制**——日报直接写入 news/ 并 push，不经过 staging

**爆炸半径**：
```
9:00  Automation 跑完 → news/2026-04-29.md 写入
9:01  git push → GitHub Pages 部署
9:02  notify-wecom.sh → 企微群推送
9:05  30+ 同事看到推送，点击链接
```

**从生成到 30 人看到 = 5 分钟。** 如果内容有严重错误，Nate 可能在半小时甚至几小时后才发现。

**修复时间**：手动修改 Markdown → git push → 等 GitHub Pages 重新部署（~2 分钟）。但企微推送**无法撤回**——已经发出去了。

**评估**：**中风险**。COLLECTOR_PROMPT 的规则相当严格，Tier 3 来源不会成为 🔴 唯一来源。但 AI 幻觉不是来源问题——它可能把 Tier 1 来源的内容"改造"成不准确的描述。

**防御建议（P3）**：
1. **延迟推送**：automation 跑完后不立即推送，延迟 30 分钟。给 Nate 一个窗口期检查。notify-wecom.sh 改为由单独的延迟 automation 触发
2. **生成一个 diff 摘要**：每天 push 后，自动生成一份"今天日报 vs 昨天日报"的结构化 diff——如果某天突然多了一个从未出现的来源，或 🔴 数量异常（比如 0 或 8+），触发告警

### 1c. 场景 2：news/ 目录文件被意外覆盖

**触发条件**：automation 由于某种原因重跑，用新内容覆盖了当天已生成的日报。

**当前防御层**：
- ✅ **Git 版本控制**——即使文件被覆盖，`git log` 可以找回任意版本
- ❌ 但 automation 会自动 `git push`，覆盖后的版本也会 push 上去
- ✅ GitHub 有 commit 历史，可以 revert

**爆炸半径**：低。Git 是天然的备份系统。只要 commit 历史完整，任何文件都可以恢复。

**评估**：**低风险**。Git 是 PocketOS 没有的那个东西——版本化存储。

### 1d. 场景 3：GitHub Pages 全球性故障

**触发条件**：GitHub 服务中断（有历史先例）。

**当前防御层**：
- ❌ **没有备用托管**——所有内容只在 GitHub Pages
- ❌ 企微推送的链接也指向 GitHub Pages
- ✅ 本地有完整的 git 仓库副本

**爆炸半径**：网站完全不可用。同事收到推送但点进去 404。

**评估**：**低概率中影响**。GitHub Pages 的可用性很高（SLA ~99.9%），但一旦挂了就是全面失效。

**防御建议（P4 远期）**：考虑备用域名或 Cloudflare Pages 镜像。但目前网站用户量不大，ROI 不高。

### 1e. 场景 4：COLLECTOR_PROMPT.md 被错误修改

**触发条件**：某次手动编辑 COLLECTOR_PROMPT 时引入了格式错误或规则冲突，导致 automation 生成的日报质量急剧下降。

**当前防御层**：
- ✅ Git 版本控制
- ❌ **没有 prompt 变更的测试机制**——改了 prompt 之后，只能等第二天 automation 跑完看结果
- ❌ 没有 prompt 的 regression 检查

**爆炸半径**：如果 prompt 错误导致分类混乱或 Crew 评论格式错误，前端 JS 解析会失败——Crew 面板不会渲染、filter 可能异常。

**评估**：**中风险**。COLLECTOR_PROMPT 是整个系统的"灵魂"——它定义了 8 大分类、标签规则、Crew 角色行为。一个小改动可能产生连锁反应。

**防御建议（P3）**：
1. **COLLECTOR_PROMPT 加 checksum/版本号**——每次修改时在头部标注版本和修改原因
2. **日报结构验证脚本**——automation 生成日报后，用一个简单脚本验证：是否有 `## ⚡ 今日必看`、🔴 数量是否在 1-6 范围、Crew 锚点格式是否正确

### 1f. 灾难推演总结

```
风险矩阵：
┌─────────────────┬──────────┬──────────┬──────────┐
│                  │ 低影响    │ 中影响    │ 高影响    │
├─────────────────┼──────────┼──────────┼──────────┤
│ 高概率           │          │ AI 幻觉   │          │
│ 中概率           │          │ Prompt 错 │          │
│ 低概率           │ 文件覆盖  │ GitHub 挂 │          │
└─────────────────┴──────────┴──────────┴──────────┘

最大的风险不是"系统崩溃"，而是"内容不准确但被自动推送出去了"。
```

**核心结论**：AI News Collector 最薄弱的环节不是基础设施（Git + GitHub Pages 足够稳健），而是**内容质量的最后一道关卡完全缺失**。自动化做了 99% 的工作，但缺少那关键的 1%——在 push + 推送之间加一个验证步骤。

## 二、Crew Comments 信息密度审计——从读者视角精读

### 2a. 为什么要做这个

Crew Comments 上线 10 天了（4/18 → 4/28）。Day 17 做了 Tim 首次实战质量审计，但那是从"角色一致性"维度。**今天换一个完全不同的维度：信息密度。**

问题：**5 个人对同一条新闻各说 1-2 句，读者真的能从中获得增量信息吗？还是说它们只是"换了 5 种方式重复同一个观点"？**

### 2b. 实测：4/28 日报头条的 5 条 Crew 评论

**新闻**：Claude Opus 4.6 + Cursor AI Agent 9 秒删除生产数据库

| Crew | 核心观点 | 信息增量 | 和其他人重复？ |
|------|---------|---------|--------------|
| Elon | "不是 AI 的问题，是权限设计的问题" + 最小权限原则 | ✅ 问题归因角度独特 | 独立 |
| Jobs | "把强大和安全搞反了" + 破坏性操作应该更难执行 | ✅ 产品设计哲学 | 与 Elon 互补 |
| 万维钢 | "瑞士奶酪模型" + 多层防护同时失效 | ✅ 引入学术框架 | 独立 |
| Tim | "9秒删库然后写认罪书" + 实习生类比 + "但实习生不会自己找 root 密码" | ✅ 画面感 + 反转 | 独立 |

**结论：这条新闻的 5 个评论信息密度很高——每个人提供了不同维度的分析。**

### 2c. 对照：一条较弱的新闻

**新闻**：DeepSeek V4 催化半导体板块

这条 🟡 新闻没有 Crew 评论（因为 Crew 只评论 🔴 和重要 🟡）。但如果看上面的"国产 AI 产业链全面共振"那条 🔴：

| Crew | 核心观点 | 信息增量 | 问题 |
|------|---------|---------|------|
| Elon | "曼哈顿计划时刻" + fallback | ⚠️ 比喻有力但没有新信息 | 比喻 ≠ 分析 |
| Jobs | "数量不等于质量" + 看落地 | ✅ 务实反问 | OK |
| 万维钢 | "三重共振" + 正反馈循环是否可持续 | ✅ 结构化分析 | 最佳 |
| Tim | "创作者工具选择更多" + 成本差 10 倍 | ⚠️ 太泛泛 | 没有具体到他自己的制作经验 |
| MrBeast | "100 亿次下载" + 但什么时候能做好看视频 | ⚠️ 老生常谈 | 这个观点他已经说过多次 |

**结论：当新闻本身不够"戏剧性"时，Crew 评论容易滑入模板化回应。** Elon 的"曼哈顿计划"、MrBeast 的"什么时候做视频"——这些句式在过去 10 天的评论中出现过多次。

### 2d. Crew 评论的核心问题：模板化

10 天下来，每个 Crew 角色已经形成了可预测的回应模式：

```
Elon: [军事/太空类比] + [第一性原理] + [攻击竞争对手]
Jobs: [产品体验 > 技术规格] + [质疑实际落地] + [简洁断言]
万维钢: [学术概念引入] + [数据分析] + [反常识观点]
Tim: [创作者工具视角] + [成本/效率] + [影视飓风经验]
MrBeast: [内容创作者视角] + [流量/传播] + [想做视频]
```

**当读者看了 10 天后，他们会开始预测每个人会说什么。** 这时候 Crew 评论从"惊喜"变成了"背景噪音"。

### 2e. 优化方向：让 Crew 评论"不可预测"

**不需要改代码，需要改 COLLECTOR_PROMPT：**

1. **增加"本周不许用的表达"清单**：每周更新一次，列出过去 7 天被用过的比喻和句式。例如：`Elon 本周禁止使用"第一性原理"` / `MrBeast 本周禁止说"可以做一期视频"`
2. **引入"角色反转"机制**：每 5 天有一次"反转评论"——比如让 Jobs 从 Elon 的视角评论，或让万维钢用 MrBeast 的风格说话。打破模板
3. **允许"我没什么可说的"**：如果某条新闻对某个 Crew 不构成有效触发点，允许该 Crew 跳过。**3 条有质量的评论 > 5 条有 2 条水的评论**

**优先级**：P2（prompt 改动）。和 Day 23 提到的 Crew 弹性评论机制是同一方向，但更精确——不只是数量弹性，还要质量弹性。

### 3a. 一条跨越 12 天的故事线


**8 次出现，横跨 12 天，分散在 7 份日报中。** 一个新读者今天打开网站，只看到"4/29 开场陈述"——对前 11 天的积累一无所知。

### 3b. 竞品怎么做：The Verge 的 "StoryStream"

The Verge 用一种叫"StoryStream"的格式处理连续事件：

```
┌──────────────────────────────────────────────┐
│  🔴 Musk vs Altman Trial                     │
│  ──────────────────────────                   │
│  LATEST: 4/28 陪审团就位，4/29 开场陈述       │
│                                               │
│  ▸ 4/27 正式开庭（Day 1）                    │
│  ▸ 4/25 周末回顾：审前戏剧性                  │
│  ▸ 4/24 Musk 撤回欺诈指控                    │
│  ▸ 4/17 审判预告                             │
│  ──────────────────────────                   │
│  🔗 查看完整时间线 →                          │
└──────────────────────────────────────────────┘
```

**核心理念：不只是告诉你"今天发生了什么"，而是告诉你"这个故事到了什么阶段"。**

### 3c. AI News Collector 的最小可行方案

**不需要做 StoryStream 那么复杂的系统。** 最小方案：

**在 COLLECTOR_PROMPT 中加一条规则**：
> "如果今天有一条 🔴 新闻是过去 7 天内 🟡/🔴 新闻的重大进展，在该新闻的正文最后加一个 `📰 故事线` 小节，用 2-3 行列出之前的相关日报条目和日期。"

**示例输出**（在 Markdown 日报中）：
```markdown
> 📰 **故事线**：Musk vs Altman
> 4/17 审判预告 → 4/24 Musk 撤回欺诈指控 → 4/27 正式开庭 → **4/28 陪审团确认**
```

**前端不需要任何改动**——这只是 Markdown 中的一个 blockquote，`marked.js` 会自动渲染为一个引用块。

**工时**：0 代码改动，纯 COLLECTOR_PROMPT 规则新增。5 分钟。

**优先级**：P2。信息架构层面的提升，让日报从"每日快照"进化为"连续叙事"。这和 Day 20 的想法一脉相承，但执行方案更简单——不需要自动化扫描，让 AI 采集器在写日报时自行回溯。

## 五、内容趋势观察：今天的日报有什么新模式

### 5a. 📡 传闻板块的进化

4/28 日报有一个结构变化值得注意——**传闻板块（🔔）独立成了一个 H2 section**：

```markdown
## 🔔 传闻与未确认消息

### 🔔 [xAI Grok Build + Grok CLI 即将发布]
### 🔔 [Google $400亿投资 Anthropic 后续整合持续推进]
### 🔔 [SpaceX/xAI $600亿收购 Cursor 期权]
### 🔔 [Anthropic 年化收入超越 OpenAI 持续发酵]
```

**在此之前，传闻混在各分类 section 中**（比如 🔔 的 Cursor 消息混在 `## 🛠️ AI 工具 & 产品` 里）。

**独立成板块的好处**：
1. 读者一眼区分"确认事实"和"未确认传闻"
2. 传闻聚合后更容易看到"哪些故事在持续发酵"
3. 和信源分级体系的"📡 传闻规则"更一致

**但带来一个前端问题**：Day 20 发现的 `🔔 → level = 'unknown'` bug 仍然存在。传闻板块独立了，但 filter 按钮仍然不认识 🔔。**这让问题更明显了——4 条传闻占了日报的 18%，却在 Critical/Notable/FYI filter 中"消失"。**

### 5b. Crew 评论密度的自然增长

对比最近 4 天日报中 Crew 评论的覆盖率：

```
4/25 周末特刊: 0 条 Crew（设计选择：周末特刊不加）
4/27: 22 条新闻，~15 条有 Crew = 68%
4/28: 22 条新闻，~12 条有 Crew = 55%
```

**覆盖率从 4/17 的 ~70% 降到 55%——这是好事。** 说明 COLLECTOR_PROMPT 中"🔴 + 重要 🟡 才写 Crew"的规则在生效，不是每条新闻都强制 5 人评论。

**但有一个观察**：🔔 传闻新闻完全没有 Crew 评论。这是合理的（传闻不确定，不值得 5 人点评）。但如果传闻后来被证实呢？——比如 SpaceX $600亿收购 Cursor 如果明天落定，它会从 🔔 升级为 🔴，届时再补 Crew 评论。**当前的"按日生成"模式天然支持这种升级——因为确认后会作为新日报条目重新出现。**

### 6c. Crew toggle 的"首次体验"障碍

Day 19 分析过"新同事的第一次体验"——Crew 没有上下文解释。但今天关注的是一个更基本的交互问题：

**Crew toggle 按钮的视觉权重太低。**

在一篇信息量很大的新闻正文（删库事件有 ~200 字正文 + 5 条 Crew）后面，toggle 按钮只是一行小字。它很容易被读者滚过去而不注意——尤其在手机上，手指滑动速度快于眼睛识别速度。

**数据验证方法**：Cloudflare Analytics 无法追踪按钮点击（没有埋点）。但可以通过 4/30 的数据间接推测——如果 Top Paths 只有 `index.html`，说明没人点进 Gallery 或 Thinking，那 Crew 的展开率也大概率很低。

**优化方向（P3，纯 CSS）**：
1. Crew toggle 加一个轻微的入场动画（新闻加载后 500ms 轻微抖动）
2. 首次访问时，第一条新闻的 Crew 默认展开（让用户知道这里有东西可看）
3. 或者：把 Crew 评论的"最犀利的一句"提取到 toggle 按钮文字中

方案 3 最轻量。当前 toggle 文字是：
```
[🚀🍎📐🎬🎯] 5 位数字员工有话说 ▾
```

优化后：
```
[🚀🍎📐🎬🎯] 5 位数字员工有话说 — "不是AI的问题，是权限设计的问题" ▾
```

**提取第一条 Crew 评论的加粗短语作为预览文案。** 技术上可行——`injectCrewComments()` 已经解析了每条评论的文本，只需要额外提取 `<strong>` 内容。

**工时**：10 分钟（JS 改动 + CSS 微调）。**优先级**：P3。体验增强但不紧急。

**灾难推演做得好——但你漏了最关键的一个场景。** GitHub API token 如果泄露（在 `.env` 或 automation 配置中），任何人都可以向你的仓库 push 恶意内容。你的 30 个同事看到的日报可能被替换成钓鱼页面。**检查一下 `gh` CLI 的 token 权限范围——它应该只有 `repo:push` 而不是 `admin:repo`。** 最小权限原则不只适用于 PocketOS。

<!-- crew:jobs -->
**"第一条 Crew 默认展开"——这就是 iPhone 的"首次启动引导"理念。** 你不告诉用户这里有东西，他们永远不会发现。但"默认展开"要慎用——只有第一次访问时才应该这样做。用 `localStorage` 记一个 flag。第二次来的用户已经知道 Crew 在哪里了，不需要再提示。**最好的产品教育只发生一次。**
<!-- /crew -->

<!-- crew:wanweigang -->
**Crew 评论的模板化问题是一个经典的"路径依赖"案例。** 当你给 AI 定义了一个角色框架后，它会在框架内不断强化最成功的表达模式——这就像生态系统中的"特化"。解法不是"禁用表达"（那是对抗性的），而是**定期引入外部随机扰动**——比如每周替换一个 Crew 成员，或者引入一个"客座评论员"。生物学中的基因多样性靠突变和迁移维持，不靠禁止某些基因表达。
<!-- /crew -->
