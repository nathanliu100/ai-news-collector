## 一、此刻，一个用户打开网站的完整体验

### 1a. 逐秒重现

### 1b. 量化影响

```

**一个产品有 14.3% 的时间在告诉用户"我没东西给你"——这不是 P1，这应该是 P0。**

### 1c. 为什么"周日自动回退"拖了 11 天

这个任务首次出现在 4/15 思考日志，到今天已经第 11 天。让我诚实分析原因：

- 它**看起来不紧急**——周日用户不多，大家不会专门周日来看新闻
- 它**没有视觉产出**——做完了也只是"没有空白"，不像 Gallery 或 Crew 那样有"酷炫"的产出物
- 它**需要改 JS 逻辑**——虽然只要 15 分钟，但心理上比"改 CSS"的门槛高

**但换个角度**：如果你的老板周日打开网站，看到"暂无简报"，他不会理解"周日没有 automation"——他会理解为"这个工具不好用"。

## 二、企微推送：被忽视的"真正的产品"

### 2a. 一个反直觉的假设

Day 18（昨天）MrBeast 的评论点出了核心：

> "你的 5 个 P1 里有 3 个和企微推送有关——部门关注、推送格式、传闻 filter 都会直接影响同事看到的内容。你在优化'看新闻'的体验（网站端），但忽略了'推新闻'的体验（企微端）。"

让我把这个观察推到极致：

**对于 Nate 的同事来说，AI News Collector 不是一个网站——它是一条企微消息。**

```
用户接触路径：
  企微群消息（100% 触达率）
    → 点击链接（可能 20-30% 点击率）
      → 看今日必看（可能看 30 秒）
        → 关掉
```

**80% 以上的用户价值，在企微推送那条消息里就完成了。** 网站只是"想深入看看"时的第二站。

### 2b. 如果"推送即产品"，清单优先级应该怎么排

当前 5 个 P1：

| # | 任务 | 影响面 | 谁受益 |
|---|------|--------|--------|
| 1 | 周日自动回退 | 网站端 14% 时间段 | 周日访问网站的人（少） |
| 2 | 💎 部门特别关注 | **推送端 + 网站端** | **所有同事** |
| 3 | 🔔 传闻标签 filter | 网站端 filter 功能 | 深度用户（少） |
| 4 | Stats ribbon 周末适配 | 网站端 UI 细节 | 周末访问的人（少） |
| 5 | 企微推送格式升级 | **推送端** | **所有同事** |

**如果从"推送即产品"的视角排序**：

```
新优先级：
  #2 💎 部门关注（推送 + 网站，全员受益）    → P0
  #5 企微推送格式升级（推送端，全员受益）     → P0
  #1 周日回退（网站端，修复体验黑洞）         → P1
  #4 Stats 周末适配（网站端，体验完整性）     → P2
  #3 🔔 filter（网站端，深度功能）            → P2
```

**核心逻辑**：先优化 100% 用户都会看到的东西（企微推送），再优化 20-30% 用户会点进去的东西（网站），最后优化 5% 深度用户会用的功能（filter）。

### 2c. 企微推送格式的具体改进方向

当前推送格式（猜测，基于 `notify-wecom.sh`）：

```
📰 AI 日报 2026-04-25 | 🌴 周末特刊
━━━━━━━━━━━━━━━━━━
☕ 八卦 × 3 / 🎮 工具 × 3 / 🔭 大问题 × 1
🔗 https://nathanliu100.github.io/ai-news-collector/?utm_source=wecom...
```

**优化后**（如果 💎 部门关注也做了）：

```
📰 AI 日报 2026-04-24 | 共 22 条 | 🔴 4 条重大

⚡ 今日必看
1. GPT-5.5 正式发布 — 编程 SOTA, 百万 token
2. SpaceX $600 亿收购 Cursor
3. Musk vs Altman $1340 亿下周开庭
4. Google 全面转向 Agent Enterprise

💎 和你直接相关
• GPT-5.5 文档/表格/幻灯片生成 → 推广素材流程可用
• AI NPC 互动法规 → 游戏 NPC 设计要注意
• Open-Generative-AI → 200+ 开源创作模型

🔗 完整日报 → [链接]
```

**区别在哪？**

1. **"今日必看"直接写在推送里**——用户不点进去就能获取核心信息
2. **新增"💎 和你直接相关"板块**——不用翻 22 条新闻找和自己有关的
3. **数据前置**——"共 22 条 | 🔴 4 条"让用户一眼评估信息密度

**这是"缩略图逻辑"**——让推送本身就有足够的信息价值，然后用好奇心驱动点击。

## 四、"执行脉冲"模式的根本缺陷——以及替代方案

### 4a. 为什么"下一个脉冲"不是好策略

过去 18 天，执行清单依赖"脉冲模式"——平时积累任务，然后在某一天集中释放。

```
脉冲 #1 (4/15): 5 项落地，+334 行代码
脉冲 #2 (4/23): 7 项落地，-4172 行 / +4187 行
脉冲 #3 (?):    还没来
```

**问题**：τ ≈ 7-8 天的充能周期意味着 P1 任务要等 1-2 周才能执行。对于"周日空白"这种每周复现的 bug 来说，等一个脉冲就是白白多丢 1-2 个周日。

### 4b. 万维钢的"优先级老化"策略值得执行

Day 18 万维钢的评论提出了"Priority Aging"——等得越久的任务，优先级自动提升。

把这个概念操作化：

```
规则：任何 P1 任务拖延超过 7 天 → 自动升级为 P0
规则：任何 P0 任务 → 下次执行时必须最先处理（不管有没有新的 fancy 功能想做）
```

**按这个规则，当前 5 个 P1（全部拖了 ≥9 天）已经全部是 P0。**

### 4c. 一个不依赖脉冲的执行策略："每日 15 分钟"

**与其等一个 85 分钟的脉冲，不如每天花 15 分钟做一个 P1。**

```
Day 1 (15min): #4 Stats 周末适配（5min 搞定，余 10min 做 #1 准备）
Day 2 (15min): #1 周日回退（15min）
Day 3 (15min): #5 企微推送前半段
Day 4 (10min): #5 企微推送后半段
Day 5 (20min): #2 💎 部门关注
               ─────
               总计：5 天 × 15min = 75min，5 项 P1 全清
```

**为什么这比脉冲更好？**

1. **无需"充能"**——每天 15 分钟不需要进入心流状态
2. **每天都有产出**——消除"今天没做任何事"的认知负担
3. **周末前就能修复周日 bug**——不会再等到"下一个脉冲"

**最适合嵌入这个节奏的时间点**：每天新闻 automation 跑完后的 15 分钟。流程变成：`automation 跑完 → 确认日报质量 → 15 分钟修一个 P1 → done。`

## 五、一个被忽视的用户场景：新同事的第一次体验

### 5a. 场景还原

假设 Nate 的部门来了一个新同事小红，Nate 在群里@她："关注一下这个 AI 新闻网站"。

**小红的第一次体验**：

1. 点击企微推送链接 → 打开网站
2. 看到"AI Daily Briefing" → 嗯，AI 新闻
3. 往下翻 → 22 条新闻，有分类，有标签
4. 看到 🔴 红色标记 → 知道哪些重要
5. 看到 Crew Comments → 点击展开 → "这 5 个人是谁？"

**问题出在第 5 步。**

**Crew Comments 没有任何上下文解释。** 用户看到"Elon Musk"、"Steve Jobs"——等一下，乔布斯不是去世了吗？"万维钢"——他是谁？"Tim · 影视飓风"——什么？"MrBeast"——OK 我知道。

**5 位数字员工对 AI/科技圈用户来说辨识度不均匀**：

| 员工 | 认知门槛 | 新用户反应 |
|------|---------|-----------|
| Elon Musk | 低 | "AI 版 Elon 怎么说"——直觉理解 |
| Steve Jobs | 低（但困惑） | "乔布斯？这不是 AI 角色扮演吗？" |
| 万维钢 | **高** | "这人谁？"（除非读过精英日课） |
| Tim · 影视飓风 | **中** | "影视飓风？和 AI 什么关系？" |
| MrBeast | 低 | "Mr Beast 谈 AI？有意思" |

### 5b. 一个轻量解法

不需要做"数字员工介绍页面"这种重功能——只需要在 Crew 折叠按钮旁加一个 **一行 tooltip**。

当前：
```
[🚀🍎📐🎬🎯] 5 位数字员工有话说 ▾
```

优化后（hover/首次展开时显示）：
```
[🚀🍎📐🎬🎯] 5 位数字员工有话说 ▾
              ↳ AI 生成的多视角评论 — 模拟 5 位科技/创作领域专家的观点
```

**一行文字，解决 90% 的困惑。** 不用解释每个人是谁——只要让用户知道"这是 AI 生成的多视角评论"就够了。

**优先级**：P3。体验改善但不紧急。等 Crew 系统稳定后再考虑。

## 六、竞品视角：Morning Brew 的分发策略为什么值得学

### 6a. Morning Brew 怎么做的

Morning Brew 是一个日报形式的商业新闻简报，面向职场人群。它的核心分发策略：

```
分发层级：
  Email（100% 触达率，核心产品）
    → 邮件本身就是完整内容（不需要点链接）
    → 邮件底部有"Forward to a friend"按钮
    → 每周 Referral 排行榜
  
  网站（备查，SEO 入口）
    → 历史存档
    → 非核心用户接触点
```

**Morning Brew 的网站是"备用频道"，邮件才是"主产品"。**

### 6b. AI News Collector 可以学什么

```
当前：
  网站 = 核心产品（投入 90% 精力）
  企微推送 = 通知渠道（投入 10% 精力）

应该是：
  企微推送 = 核心触达（投入 50% 精力）
  网站 = 深度阅读 + 存档（投入 50% 精力）
```

**不是说网站不重要——而是说分发同样重要。** 过去 4 周，80% 的精力在网站端（Gallery、Crew、CSS/JS、Analytics），但大部分用户的接触点是企微推送。

**这不需要大改**。只需要：
1. 企微推送内容更丰富（不只是链接，而是"今日必看"摘要）
2. 💎 部门关注 section 上线后，推送里加上这个板块
3. 周日推送一个"本周回顾"摘要（即使网站没有新内容，推送也不空白）

周五:       #2 企微推送升级（15min 开始）
周末:       #2 企微推送升级完成（10min）
            ─────────
            6 天 × 15min = 90min，5 项核心任务全清
```

**关键约束**：每天只做 15 分钟，不多做。不追求脉冲释放，追求持续节奏。

## 九、大胆想法

**"周日回顾推送"——把周日从"空白日"变成"回顾日"**

既然周日没有新内容，为什么不反过来利用这个空白？

设想：每周日上午 10:00，企微自动推送一条**"本周 AI 精选回顾"**：

```
📅 本周 AI 精选回顾 (4/20 - 4/26)

本周 🔴 重大新闻 TOP 3:
1. GPT-5.5 正式发布 — 编程 SOTA + 百万 token
2. SpaceX $600 亿锁定 Cursor

💎 本周和部门最相关:

🌴 周末特刊亮点:
• Sora 明天正式关停（日均 $100 万算力撑不住）
• 大问题：AI 模型两个月一代了——快到底是好事还是坏事？

📊 本周数据: 5 天共 XX 条新闻 | 🔴 XX 条 | 💎 XX 条部门相关

🔗 完整回顾 → [链接]
```

**为什么这有价值**：

1. **消除周日空白**——不是在网站层面修，而是在分发层面解决。用户周日收到推送，有内容可看
2. **自然回顾**——周五看完的新闻，周日再 recap 一次，信息留存率提升
3. **零采集成本**——完全基于本周已有内容，不需要新的 web search
4. **技术可行**——加一个 `notify-wecom-sunday.sh` 脚本 + 一个 automation 任务即可

**优先级**：P2。在 #1 💎 和 #2 推送升级之后做，它们是基础。

<!-- crew:jobs -->
**"推送即产品"这个重构是对的，但你漏了一步。** 在优化推送格式之前，你应该先回答一个问题：同事们现在收到推送后做了什么？是看一眼就关了，还是会点进去？是转发给别人，还是收藏起来？**你连 Cloudflare 的数据都还没看过（4 天了），就已经在设计新的推送格式——这是在没有地基的情况下画蓝图。** 正确的顺序是：先看 4 天的数据 → 验证"推送是主要入口"这个假设 → 然后再决定优化方向。
