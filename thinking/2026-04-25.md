### 1b. "大问题"这个板块正在成为周末特刊的灵魂

本期大问题："AI 模型两个月一代了——快到底是好事还是坏事？"

这篇写得很好。几个亮点：

1. **引用了红皇后效应（Red Queen Effect）**——Lewis Carroll 的文学典故 + 生物学概念，有万维钢的味道
2. **用数据说话**——GPT-5.4 到 5.5 不到两个月，Kimi K2.6 开源首超闭源
3. **结尾落到部门实操**——"不要追着最新模型跑，选一个够用的工具深度嵌入工作流"

**但有一个可以改进的地方**：这篇文章的论点——"模型迭代太快，产品化跟不上"——和 4/23 日报的编辑观察高度重合。周末特刊的"大问题"应该**比日报的编辑观察更深入一个层次**，而不是把同样的观点用更长的篇幅复述一遍。

**改进方向**：大问题应该做到日报编辑观察做不到的三件事：
- **跨越一周时间线**：不是某一天的观察，而是这一周 5 天新闻的元模式
- **引入外部知识**：不只是 AI 圈的事实，还要链接到其他学科（这期的红皇后效应做到了）
- **留下一个开放问题**：不给结论，给思考框架。让读者带着问题进入下一周

### 1c. 周末特刊的产品定位正在清晰化

两期下来，模式已经形成：

```
☕ 八卦 = 周一同事茶水间聊 AI 的谈资（轻松、有梗、1 分钟能读完）
🎮 工具 = "这周末我试试什么"的行动清单（每个工具都有 5 分钟上手指南）
🔭 大问题 = 深度思考（万维钢式，800-1200 字）
📊 数据快照 = 这周发生了什么的事实锚点
```

**这四个板块的角色分工很健康。** 但我注意到一个缺失——**周末特刊没有 Crew 评论**。

日报有 Crew 评论，Gallery 有 Crew 评论，唯独周末特刊没有。这是有意的选择还是遗漏？

**分析**：
- **不加 Crew 的理由**：周末特刊是"轻松阅读"，5 人评论会让它变重
- **加 Crew 的理由**："大问题"板块特别适合 Crew 评论——800 字的深度分析后面跟着 5 个不同视角的反应，信息密度极高

**建议**：**只在"大问题"板块加 Crew 评论，其他三个板块不加。** 这样周末特刊保持轻松的整体调性，但最核心的深度内容有 Crew 加持。

**优先级**：P2。需要修改 `WEEKEND_PROMPT.md`。

## 二、Sora 关停 × Gallery 策略的"保鲜期"问题

### 2a. 一个具体的场景

今天的周末特刊八卦第一条："Sora 明天正式死亡，算力账单日均 100 万刀。"

Gallery 里目前有 2 个作品，其中一个使用了 Higgsfield 技术。如果 Higgsfield 明天也宣布关停呢？那这个 Gallery 作品就变成了一个"历史文档"而不是"可复现的案例"。

**这不是假设——而是正在发生的现实。** Sora 从惊艳全网到关停只用了 2 年。AI 视频工具的生命周期可能比我们想象的短得多。

### 2b. Gallery 的"保鲜"策略

当前 Gallery 的作品卡片信息结构：
```
作品名 + 视频 + 工具 + 简介 + Crew 评论
```

**缺失的关键字段：制作日期 + 工具版本 + 工具现状（活跃/已关停）**

如果加上这些，Gallery 就从"作品展示"变成了"AI 工具效能编年史"——即使工具关停了，作品仍然有**基准参考价值**："2026 年 4 月用 Higgsfield 能做到这个水平"。

**这和 4/22 思考日志提出的"AI 工具效能档案"是同一个方向，但 Sora 关停让它有了紧迫性。**

### 2c. 一个更大的产品思考

Sora 的死亡给了 Gallery 一个意外的定位机会：

```
当前定位：展示"最新 AI 工具能做什么"（工具导向）
更好的定位：记录"AI 影像创作的能力边界如何随时间推移"（编年史导向）
```

**区别在哪？**

工具导向 = 当工具关停时，作品失去意义
编年史导向 = 当工具关停时，作品获得更多意义——它成为那个时代的能力快照

Morning Brew 不会因为某个公司倒闭就删掉当年的报道。Gallery 也不应该因为某个工具关停就隐藏作品。

**但需要前端支持**：给关停工具的作品卡片加一个视觉标记——比如一个低饱和度的"Archive"徽章，表示"这个工具已经不可用了，但作品仍然有参考价值"。

**优先级**：P3 愿景。先记在这里，等 Gallery 数据化（JSON + 模板）完成后再考虑。

## 四、一个被忽视的体验问题：周末用户进网站看到什么？

### 4a. 重现用户路径

假设 Nate 的同事今天周六收到企微推送（`notify-wecom-weekend.sh`），点击链接进入网站。他看到什么？

**当前行为**：
1. 打开 `index.html`
2. `loadNews()` 尝试加载 `2026-04-25-weekend.md`（周末特刊优先）
3. ✅ 成功！显示周末特刊内容
4. Stats ribbon 显示……什么？

**问题就在 Stats ribbon。**

Stats ribbon 统计的是 `🔴 🟡 🟢` 的数量——但周末特刊没有这些标签。周末特刊的结构是 `☕ 🎮 🔭 📊`，不是 `🔴 🟡 🟢`。

**结果**：Stats ribbon 在周末特刊页面上会显示全是 `—`（空值），因为 `updateStats()` 函数匹配的是 `### 🔴` 等模式，周末特刊里没有这些。

**这就是清单里的 #4「📊 Stats ribbon 周末适配」——已经拖了 11 天的 P1 任务。**

### 4b. 修复方案（5 分钟的事）

两个方向：
- **方案 A：周末特刊时隐藏 Stats ribbon** — 简单粗暴，`display: none`
- **方案 B：适配周末格式** — 识别 `☕ 🎮 🔭 📊` 模式，显示"3 八卦 · 3 工具 · 1 大问题"

**方案 B 更好**——它给周末特刊自己的身份感，而不是简单地"没有统计"。

```
工作日 Stats: 共 22 | 🔴 4 | 🟡 11 | 🟢 3 | 📰 8 | 𝕏 4 | 🏢 10
周末 Stats:   ☕ 3 八卦 · 🎮 3 工具 · 🔭 1 大问题
```

**这个改动可以在 `app.js` 的 `updateStats()` 函数里加一个分支判断——检测 markdown 里有没有 `## ☕`，如果有就是周末模式。**

### 4c. 更深的问题：周日怎么办？

明天是周日。周日没有新闻采集（不是工作日），也不是周末特刊日（特刊是周六发）。

用户周日打开网站 → `loadNews()` 尝试加载 `2026-04-26.md` → 404 → 显示"暂无简报"。

**这就是清单里的 #1「📅 周日自动回退」——已经拖了 9 天的 P1 任务。** 如果明天不修，就是**连续第 3 个周日空白**。

**修复方案**：
- `loadNews()` 404 后，回退尝试 `当天-1`、`当天-2`、`当天-3`
- 加一个轻量 banner："今天是周日，显示最近一天的新闻"
- 15 分钟搞定

**三个周日空白了。再拖就说不过去了。**

## 五、竞品观察：从 Sora 关停看"AI 新闻产品"的生命周期

### 5a. Sora 的一生

```
2024-02: 发布，全网惊艳
2024-12: 公测，评价两极
2025 全年: 持续投入，成本居高不下
2026-04-26: 关停，日均 100 万刀算力撑不住
```

**Sora 不是因为不好而死，是因为成本结构不可持续。**

### 5b. AI News Collector 的成本结构

让我们诚实地算一下 AI News Collector 的"运营成本"：

```
每日成本：
  - WorkBuddy automation 跑一次（API token 消耗）~= ？
  - GitHub Pages 托管 = $0
  - Cloudflare Analytics = $0
  - Cloudflare R2 视频托管 = 接近 $0（出站流量免费）
  - 企微推送 = $0（Webhook 免费）

唯一的非零成本 = AI API 调用
```

**和 Sora 对比，AI News Collector 的成本结构非常健康。** 没有实时推理、没有大规模算力、没有用户端服务器。唯一的成本是每天一次的新闻采集 automation。

**但有一个隐性成本：Nate 的时间。** 过去 18 天，Nate 在网站上投入了大量精力——不是写代码的时间（那是自动化的），而是**思考、决策、审核内容质量**的时间。这个隐性成本比 API 费用高得多。

**产品启示**：网站需要逐步减少 Nate 的人工介入。目前需要 Nate 人工参与的环节：
1. ❌ 新闻采集 → 已自动化 ✅
2. ❌ 日报发布 → 已自动化 ✅
3. ❌ 企微推送 → 已自动化 ✅
4. ⚠️ 周报写作 → 半自动化（需要 Nate 审核方向）
5. ⚠️ Gallery 内容 → 完全手动（Nate 挑选 + 填写作品信息）
6. ⚠️ 思考日志 → 自动化但需要持续校准

**Gallery 是唯一没有自动化路径的环节**——这也是为什么 Gallery 只有 2 个作品。


**为什么这个想法有价值**：

## 九、Cloudflare Analytics Day 3：该看什么了

### 9a. 三天够了吗？

Day 1（周三下午接入）= 半天数据
