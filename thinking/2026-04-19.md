### 2. 周末体验黑洞——一个比想象中更严重的问题

> P0 周末路径 bug 已经存在 2 天。但这只是冰山一角——整个"周末"场景从未被认真设计过。

#### 2a. 周末用户的完整体验链

假设一个 Nate 的同事在周日下午 3 点打开网站：

```
Step 1: 访问 https://nathanliu100.github.io/ai-news-collector/
Step 2: 默认加载今天的日期 (2026-04-19)
Step 3: fetch('news/2026-04-19.md') → 404
Step 4: fallback fetch('news/2026-04-19.en.md') → 404
Step 5: 显示"暂无简报"空状态
Step 6: 用户困惑——点击 ← 箭头回到 4/18
Step 7: fetch('news/2026-04-18.md') → 404（因为实际文件是 2026-04-18-weekend.md）
Step 8: 显示"暂无简报"空状态
Step 9: 用户再次困惑——继续点 ← 到 4/17
Step 10: fetch('news/2026-04-17.md') → 200 ✅ 终于看到内容了
```

**用户需要点击 2 次"上一天"才能看到任何内容。** 而且在这个过程中，他们完全不知道周末特刊的存在。

这不是一个 edge case——**每周都会有两天（周末）这个体验是坏的。** 也就是说 28.6% 的时间，产品的首屏体验是"空白"。

#### 2b. 但 bug 之外，还有一个更根本的问题

即使修了路径 bug（让 4/18 能加载 `-weekend.md`），4/19（周日）仍然是空的——因为根本没有周日的新闻。

**这意味着周日的体验永远是空白的（除非周末特刊也覆盖周日）。**

解决方案梯度：

| 方案 | 难度 | 效果 | 推荐？ |
|------|------|------|--------|
| A. 修路径 bug + 周日自动回退到最近有内容的日期 | 15min | 用户打开就有内容 | ✅ **最推荐** |
| B. 周末特刊覆盖周六+周日两天 | 改 automation 命名规则 | 需要重新设计文件名 | 可以但复杂 |
| C. 周日也跑 daily automation | 改 cron 调度 | 增加内容，但周日 AI 圈新闻很少 | 性价比低 |
| D. 保持现状 | 0 | 28.6% 时间空白 | ❌ |

**方案 A 的具体实现**：

```javascript
// 在 loadNews() 404 后，不是显示空状态，而是自动回退到最近的有效日期
async function loadNews(dateStr) {
    // ... 现有 fetch 逻辑 ...
    if (!res.ok) {
        // 尝试 weekend 版本
        res = await fetch('news/' + dateStr + '-weekend' + langSuffix);
    }
    if (!res.ok) {
        // 自动回退：往前找最近 3 天
        for (var offset = 1; offset <= 3; offset++) {
            var fallbackDate = new Date(dateStr + 'T00:00:00');
            fallbackDate.setDate(fallbackDate.getDate() - offset);
            var fbDateStr = fmtDate(fallbackDate);
            // 尝试普通版 + weekend 版
            res = await fetch('news/' + fbDateStr + langSuffix);
            if (!res.ok) res = await fetch('news/' + fbDateStr + '-weekend' + langSuffix);
            if (res.ok) {
                // 更新 UI 显示"展示的是 X 日的内容"
                showFallbackNotice(fbDateStr);
                break;
            }
        }
    }
}
```

**关键 UX 细节**：回退时必须告诉用户"你看到的是 4/18 的内容"，否则用户会以为 4/19 的新闻就是这些。一个简单的 banner：

```
📅 今天（周日）无新闻更新，为你展示 04/18 周末特刊
```

**工时**：20min（包含 fallback 逻辑 + banner UI）。**建议合并到 P0 周末路径 bug 一起修。**

#### 2c. History Panel 也有周末盲区

`scanFiles()` 函数只扫描 `news/YYYY-MM-DD.md` 格式的文件：

```javascript
var r = await fetch('news/' + ds + '.md', { method: 'HEAD' });
```

这意味着 History Panel 里看不到 `2026-04-18-weekend.md`。用户打开存档，4/18 那天会显示为"不存在"。

**修复**：在 scan 循环里额外 HEAD 一下 `${ds}-weekend.md`。5min。

#### 3b. "拖延成本"的三个层面

**层面 1：直接用户影响**
- P0 路径 bug：每个周末来访的用户都受影响。假设每天 10 人访问，2 天 = 20 次"空白体验"
- P1 🔔 filter bug：每天的传闻新闻（2-4 条）不被计数或筛选

**层面 2：信任损耗**
- 如果 Nate 给同事推荐这个网站，同事周末点进去看到空白，Nate 的推荐信用就损耗了
- 企微推送缺少信源标签 → 同事不知道新闻来源是否可靠

**层面 3：技术债复利**
- CSS/JS 分离每天不做，新增的每一行代码都让分离难度增加一点
- 2208 行现在分离大概 30min。再加 500 行（第 4 周的预期增量），可能需要 45min
- 这就是技术债的"复利效应"——越晚做，成本越高

## 💡 今天的大胆想法

**"周末模式"——网站应该有两张面孔**

当前问题的根源是：网站被设计成一个"工作日日报查看器"，但它需要在一周七天都能用。

设想：**网站检测当前是工作日还是周末，自动切换 UI 模式**。

```
工作日模式（周一-周五）：
├── 默认加载当天日报
├── 22 条新闻 + 8 分类
├── Stats ribbon 显示计数
├── Filter 按钮（🔴🟡🟢）
└── Crew Comments 在新闻下

周末模式（周六-周日）：
├── 默认加载最新的周末特刊
├── 顶部 banner："📅 Weekend Edition"
├── 不显示 Stats ribbon（周末特刊无分类计数）
├── 不显示 Filter 按钮（周末特刊不需要过滤）
├── 底部"快速回顾"链接：本周 Mon → Fri 的日报
└── 更宽松的排版（行距 +10%，配合轻松阅读节奏）
```

**为什么这不只是"修一个 bug"这么简单**：

路径 bug 的修复是"让周末能看到内容"。但"周末模式"是**让周末的体验被专门设计**。两者的差别就像"修好门锁"和"重新设计入口"的区别。

**技术实现非常简单**：

```javascript
var today = new Date();
var isWeekend = (today.getDay() === 0 || today.getDay() === 6);

if (isWeekend) {
    document.body.classList.add('weekend-mode');
    document.getElementById('statsRibbon').style.display = 'none';
    // 自动加载最新周末特刊
    loadWeekendEdition();
} else {
    loadNews(fmtDate(today));
}
```

**但优先级不高**——P3 冰箱级。先修好路径 bug（P0），等周末特刊稳定运行 2-3 周后再考虑专门的周末模式。
