### 1. P0 修复效果验证——代码审视

> 周末路径 bug 存活了 3 天（4/18-4/20 中午），今天终于修了。但修得怎么样？

#### 1a. `newsPathCandidates()` 的 fallback 链

审读代码（index.html L1699-1711），当前实现了一个 4 级候选路径链：

```
周末日期：
  1. news/YYYY-MM-DD-weekend.{lang}.md   ← 优先加载周末版+当前语言
  2. news/YYYY-MM-DD-weekend.{fallback}.md ← 周末版+另一语言
  3. news/YYYY-MM-DD.{lang}.md            ← 普通日报+当前语言
  4. news/YYYY-MM-DD.{fallback}.md         ← 普通日报+另一语言

工作日日期：
  1. news/YYYY-MM-DD.{lang}.md
  2. news/YYYY-MM-DD.{fallback}.md
```

**评价：✅ 设计合理。** 周末优先 weekend 版，fallback 到普通日报，再 fallback 到另一语言。4 层候选覆盖了所有常见场景。

**但有一个遗漏**：4/19 日志提出的"自动回退到最近有效日期"功能**没有实现**。

现状：4/19（周日）没有任何新闻文件——既没有 `2026-04-19-weekend.md`（只有 4/18 有周末特刊），也没有 `2026-04-19.md`（周日不跑 daily automation）。4 个候选路径全部 404，用户看到"暂无简报"。

**这意味着每周日的体验仍然是空白的。**

| 日期 | 文件是否存在 | 用户体验 |
|------|------------|---------|
| 周一-周五 | ✅ `YYYY-MM-DD.md` | 正常显示日报 |
| 周六 | ✅ `YYYY-MM-DD-weekend.md` | ✅ 修复后正常显示周末特刊 |
| **周日** | ❌ 无文件 | **⚠️ 仍然空白** |

**修复方案**（重申 4/19 的建议）：

```javascript
// 在 loadNews() 所有候选都 404 后，自动回退到最近 3 天
if (!res) {
    for (var offset = 1; offset <= 3; offset++) {
        var fb = new Date(dateStr + 'T00:00:00');
        fb.setDate(fb.getDate() - offset);
        var fbStr = fmtDate(fb);
        var fbCandidates = newsPathCandidates(fbStr);
        for (var j = 0; j < fbCandidates.length; j++) {
            var tryRes = await fetch(fbCandidates[j]);
            if (tryRes.ok) { res = tryRes; showFallbackBanner(fbStr); break; }
        }
        if (res) break;
    }
}
```

**工时**：15min（含 banner UI）。**优先级**：P1——不影响工作日用户，但每周日都有一天空白。

### 2. Crew Comments 实战审视——从 4/20 日报看效果

> 4/20 日报是 Crew Comments 写入 COLLECTOR_PROMPT 后的**首份自动化生成**（4/18 的 crew 是手动补写的，今天是 prompt 驱动自动生成的第一天）。

#### 2a. 覆盖率分析

4/20 日报 20 条新闻中，Crew Comments 的分布：

| 新闻级别 | 条数 | 有 Crew 评论 | 覆盖率 |
|---------|------|-------------|--------|
| 🔴 重大 | 4 | 4（Opus 4.7、Claude Design、Canva AI 2.0、Cursor） | **100%** |
| 🟡 关注 | 10 | 0 | 0% |
| 🟢 了解 | 3 | 0 | 0% |
| 🔔 传闻 | 3 | 0 | 0% |

**评价：覆盖率符合预期。** COLLECTOR_PROMPT 指令是"每条 🔴 重大 + 重要 🟡 新闻必须写"，但实际只覆盖了 🔴。10 条 🟡 新闻全部无 crew 评论。

**这引发两个问题：**

1. **🟡 新闻是否真的需要 crew？** 从用户体验看：🔴 4 条 × 5 个 crew = 20 段评论，已经是不小的内容量。如果 🟡 也加，会变成 (4+10) × 5 = 70 段评论——信息爆炸。**建议维持现状：只给 🔴 加 crew，🟡 选择性加（比如带 💎 标签的 🟡）。**

2. **Crew 评论的质量如何？** 逐条审视 4/20 的 crew 表现：

| Crew | 角色一致性 | 内容质量 | 有没有新信息 |
|------|-----------|---------|------------|
| Elon | ✅ 战略+商业视角（"在倾销"、"定价权被撬"） | ⭐⭐⭐⭐ | ✅ "价格倾销"角度是新闻正文没提到的 |
| Jobs | ✅ 产品/用户体验（"功能在膨胀，叙事在稀释"） | ⭐⭐⭐⭐⭐ | ✅ 一针见血指出 Anthropic 同时发 3 个新概念的问题 |
| 万维钢 | ✅ 数据实证（"58→70 是+12 个百分点"） | ⭐⭐⭐⭐ | ✅ 区分了 SWE-bench 和 CursorBench 不能混着看 |
| Trump | ⚠️ 浮夸但稍显重复 | ⭐⭐⭐ | ❌ "非常好""相信我"模板感偏强 |
| MrBeast | ✅ 内容创作者视角（"一眼判断封面能不能上热门"） | ⭐⭐⭐⭐ | ✅ 给出了具体的使用场景想象 |

**总评**：Jobs 和万维钢的评论质量最高，每条都有新增信息量。Trump 有 template 化倾向（"非常好""最好的""相信我"出现频率太高），建议在 COLLECTOR_PROMPT 里加一条：**"Trump 不应每条都用'非常好/最好的'，偶尔可以反对或质疑"**。

#### 2b. 一个微妙但重要的 UX 发现

阅读 4/20 日报的 Markdown 源码，发现 crew 评论紧跟在新闻正文之后、下一个 `###` 标题之前：

```markdown
### 🔴 [Claude Opus 4.7 正式发布](...)
**来源**：...
正文段落...
<!-- crew:elon -->评论内容<!-- /crew -->
<!-- crew:jobs -->评论内容<!-- /crew -->
...

### 🟡 [下一条新闻](...)
```

在 `index.html` 的 `structureNewsBlocks()` 中，H3 标签后的所有内容直到下一个 H3/H2/HR 都被包进同一个 `.news-block` div。这意味着 crew 的 `<span class="crew-data">` 也在 `.news-block` 内——`injectCrewComments()` 能正确找到并解析它们。

**但有一个边缘情况**：如果 crew 评论的 Markdown 中包含 `###`（H3 标题格式），`structureNewsBlocks()` 会提前断开 block，导致后面的 crew 评论被孤立到 block 外。

**当前风险**：低。因为 crew 评论是 1-2 句加粗文字，不包含 H3。但如果未来 crew 评论格式复杂化（比如包含代码块或列表），需要注意这个 parser 边界。

#### 2c. Crew Comments 的数据量影响

对比有 crew 和没有 crew 的日报文件大小：

```
4/17 日报（有 crew，手动补写的前一版）：~8.5KB
4/20 日报（有 crew，prompt 自动生成）：  ~9.2KB
假设无 crew 的 4/20 版本：              ~7.0KB（估算）
```

**Crew 评论约占总文件大小的 24%**。这意味着：
- 对网络加载影响微乎其微（+2KB）
- 对阅读体验影响取决于折叠交互——默认折叠 ✅ 不增加首屏信息负担
- 对采集 prompt 的 token 消耗有一定增加（每条 🔴 新闻多生成 ~200 tokens × 5 crew）

当一天的 🔴 新闻中有 2 条以上带 💎 时，可以认为这是一个"黄金日"——对部门有重大影响的日子。

设想一个自动化反应链：

```

  ↓
企微推送格式自动切换为"💎 优先"模式
  ↓
推送标题加前缀："⚠️ 部门重点关注日"
  ↓

```

**工时**：推送格式切换 ~10min（notify-wecom.sh 加条件判断）；首屏展开 ~5min（JS 改一个 auto-open 逻辑）。

**优先级**：P3 冰箱级。先把 💎 section 本身做出来，再考虑自动化联动。

## 💡 今天的大胆想法

**"新闻冲击波"——当 AI 新闻直接影响你的工作时，网站应该主动告诉你**

今天的核心洞察：4/20 是一个"黄金日"——4 条 🔴 中 2 条直接冲击设计团队。但网站把它们和其他新闻一样平等展示，没有任何"这条和你特别相关"的信号。

设想一个"冲击波"功能：

```
用户打开网站，首屏顶部出现一个红色横幅：

  ┌─────────────────────────────────────────────────┐
  │  ⚠️ 今日有 2 条新闻直接影响你的工作             │
  │  🔴 Canva AI 2.0：你正在用的设计工具大改了        │
  │  🔴 Claude Design：新竞品入场，设计流程要重评估    │
  │                                    [查看详情 ↓]   │
  └─────────────────────────────────────────────────┘
```

**这不是信息筛选，这是信息预警。**

Morning Brew 不会做这个——因为它面向所有人。Hacker News 不会做——因为它靠投票排序。但 AI News Collector 知道它的用户是谁（IEG 游戏多媒体设计）。**这是小产品的天然优势：你认识你的用户。**

技术实现分两层：
1. **采集 prompt 层**：已有 💎 标签，只需在 Markdown 头部增加一个机器可读的 meta 字段：`<!-- impact: high, targets: design,3d -->` → 15min
