### 1. Gallery 功能——从信息聚合到作品展示

> `gallery-mockup.html` 是 AI News Collector 的第三个独立页面（主站 `index.html` + Dev Lab `thinking/index.html` + Gallery）。它代表了产品的一个重要方向转变。

#### 1a. Gallery 做了什么

从代码审读（843 行独立 HTML）：

- **定位**：AI 视频创作 Gallery，展示 Nate 和团队用 AI 工具制作的视频作品
- **设计语言**：继承主站 CSS 变量体系（same `--bg`, `--accent`, `--mono` 等），保持品牌一致性
- **布局**：Hero banner + 筛选栏（Filter chips）+ 网格展示
- **导航**：主站 Mode Tabs 新增第三个 tab "🎬 Gallery"，用 `<a>` 标签直接链接到独立页面
- **暗色模式**：已支持（继承 `prefers-color-scheme: dark`）

#### 1b. 产品价值分析——为什么 Gallery 比看起来更重要

退一步看：AI News Collector 最初的定位是**"信息消费工具"**——帮 Nate 的同事了解 AI 前沿。Gallery 的加入，让它开始变成**"信息消费 + 创作展示"**的混合体。

这个转变很有意义：

```
之前的价值主张：
  "每天看 AI 新闻" → 被动消费 → 用户留存靠习惯

加入 Gallery 后：
  "每天看 AI 新闻 + 看团队用 AI 做了什么" → 消费 + 灵感 → 用户留存靠价值认同
```

**关键洞察**：对 Nate 的设计师同事来说，"看到别人用 AI 工具做出了什么东西"的吸引力，可能远大于"又发了一个新模型"。Gallery 把抽象的"AI 趋势"变成了具体的"AI 产出物"——这正是设计师群体需要的那根连接线。

**竞品对比**：Morning Brew、TLDR、The Rundown 都没有 Gallery。它们只做"告诉你 AI 有什么新的"，不做"展示 AI 能做什么"。**Gallery 是 AI News Collector 成为团队内部工具而非纯新闻站的关键差异点**。

#### 1c. Gallery 体验审视

**做得好的：**

1. **独立页面而非嵌入主站**——正确选择。Gallery 的展示逻辑（网格 + 筛选 + 视频播放）与日报的阅读逻辑完全不同，强行塞到 `index.html` 里只会让 2246 行变成 3000+ 行
2. **CSS 变量复用**——字体、颜色、间距全部用了同一套 `--` 变量。切换页面时视觉连贯，不像两个不同的网站
3. **筛选栏设计**——Filter chips 的视觉风格和主站的 pill 按钮一脉相承

**需要改进的：**

**问题 1：Mode Tabs 中 Gallery 是唯一的 `<a>` 标签——跳转体验断裂**

```html
<button class="mode-tab active" data-mode="daily" onclick="setMode('daily')">Daily</button>
<button class="mode-tab" data-mode="weekly" onclick="setMode('weekly')">Weekly</button>
<a class="mode-tab" href="gallery-mockup.html" data-mode="gallery">🎬 Gallery</a>
```

Daily 和 Weekly 切换是 **SPA 式**的（同一个页面内切换内容），Gallery 是**页面跳转**。用户的心理模型是"三个 tab，点任何一个都是即时切换"——但 Gallery 点击后整个页面重新加载。

**体验断裂点**：
- 点击 Gallery → 全屏闪白（页面重载）→ 新页面打开
- 从 Gallery 想回 Daily → 没有返回入口？（需要审视 gallery-mockup.html 的导航）

**建议**：给 Gallery 页面顶部加一个返回主站的导航（或者复制主站的 mode-tabs header），让用户能无缝回来。工时 ~15min。

**问题 2：文件命名——`gallery-mockup.html` 的 "mockup" 暗示临时性**

"mockup" 这个词在设计语境里意味着"还不是最终版"。如果这个页面要长期存在，建议改名为 `gallery.html`。这也影响 SEO 和 URL 美观度。

**工时**：5min（改文件名 + 更新主站链接）。

**问题 3：从 Gallery 返回主站的路径未知**

Gallery 页面（`gallery-mockup.html`）是否有返回主站的导航？如果没有，用户到了 Gallery 就"出不来"了——只能手动修改 URL 或按浏览器返回。

**这是一个 P2 的 UX 问题**。

#### 1d. Gallery 引发的架构反思

现在 AI News Collector 有 **3 个 HTML 文件**：

```
index.html          2246 行   主站（日报 + 周报 + 新闻查看器）
gallery-mockup.html  843 行   Gallery（AI 创作展示）
thinking/index.html  ???  行   Dev Lab（产品思考 + changelog）
                    ─────────
                    ~3500+ 行 HTML 总量
```

**这是一个多页面应用（MPA），不是单页应用（SPA）。** 但它在装作是 SPA——Mode Tabs 看起来像 tab 切换，实际上两个是 JS 切换、一个是页面跳转。

这不算错，但随着页面增多，会面临两个问题：

1. **CSS 重复**：Gallery 重复定义了主站的所有 CSS 变量和基础样式（~200 行）。如果要改主题色，得改 3 个文件
2. **导航不一致**：主站有完整的 nav-bar（日期导航 + filter + 语言切换），Gallery 有自己的 nav，Dev Lab 有自己的 nav。三个页面的导航体验不一致

**这强化了一个已经喊了 15 天的需求——CSS/JS 分离**。现在不只是"单文件太长"的可维护性问题，而是"多文件间样式重复"的效率问题。CSS 分离出 `style.css` 后，三个页面可以共用同一份样式。

**CSS/JS 分离的优先级应该升至 P0。** 理由：
1. 不再只是可维护性——现在是多页面间的一致性问题
2. 每新增一个页面（比如未来可能的 podcast 页、dashbaord 页），重复成本都在累加
3. Gallery 的 843 行中，估计 200+ 行是和主站重复的 CSS

#### 3b. 为什么总是清单外的东西赢

回到万维钢的 Present Bias 分析（4/18 日志）——但我想换一个框架来理解这个现象。

心理学家 Mihaly Csikszentmihalyi 的**心流理论**说：人在"挑战适中 + 技能匹配 + 即时反馈"时进入心流状态。对比一下：

| | 清单任务（如 CSS/JS 分离） | 灵感任务（如 Gallery） |
|---|---|---|
| 挑战感 | 低（纯机械拆分） | 高（新页面设计 + 交互） |
| 即时反馈 | 弱（"文件拆了"，没有视觉变化） | 强（做完就能看到一个新页面） |
| 心流触发 | ❌ | ✅ |

**CSS/JS 分离是一个"正确但无聊"的任务**——它不会让产品看起来更好，只会让下一次修改更快。这种延迟回报的任务，在没有外部压力（deadline / 老板 / 生产事故）的情况下，几乎不可能胜过"做一个新 Gallery"的即时满足。

**解决方案**（不是靠意志力，是靠系统设计）：

1. **捆绑策略**：把 CSS/JS 分离和 Gallery 绑定——"Gallery 页面需要复用主站样式 → 先抽出 style.css → 再让 Gallery 引用它"。这样分离就不再是"纯机械操作"，而是"为了让 Gallery 更好而必须做的事"
2. **可视化清单**：每天在思考日志开头用一个"进度条"展示清单完成率。当视觉上看到进度条不动，心理压力会自动增加

#### 3c. 执行清单 v14——接纳现实后的重新排序

现实是：Nate 更可能做"有趣的新东西"而不是"修老 bug"。那清单就应该反映这个现实——**把"有趣但也重要"的任务排在前面**。

## 💡 今天的大胆想法

**"AI 工具效能档案"——Gallery 不只是展示作品，而是记录每个作品背后的 AI 工具链**

灵感来源：4/21 新闻密集出现了 AI 工具的定价和性能数据（Cursor $500 亿、Canva AI 2.0、Claude Design、可灵 3.0 $6.99/月），但这些数据对设计师来说**太抽象了**。

设想 Gallery 的每个作品卡片不只有视频/图片，还有一个**"工具链"标签**：

```
┌─────────────────────────────────────────┐
│  🎬 POE2 短剧 · 荒野之息                 │
│  [视频缩略图]                            │
│                                         │
│  🛠️ 工具链                              │
│  ├── 剧本: Claude Opus 4.6              │
│  ├── 分镜: Midjourney v7                │
│  ├── 视频: Kling 3.0                    │
│  ├── 配音: ElevenLabs                   │
│  └── 剪辑: Premiere + Runway Gen-4.5    │
│                                         │
│  ⏱️ 总耗时: 4 小时                       │
│  💰 总成本: ¥180                         │
│  📅 制作日期: 2026-04-15                  │
│                                         │
│  📊 同类作品传统制作估价: ¥15,000+         │
│     → AI 效能倍率: 83×                   │
└─────────────────────────────────────────┘
```

**为什么这有杀伤力**：

1. **量化了 AI 的"真实生产力"**——不是 benchmark 分数，是实际节省的时间和金钱
2. **消除了同事的 AI 焦虑**——看到"4 小时 + ¥180 做了一个以前要 ¥15000 的东西"，好奇心 > 恐惧
3. **形成了历史纵深**——半年后回头看，工具链从"Kling 2.0 → 3.0"，成本从"¥300 → ¥180"，趋势一目了然
4. **和 AI News Collector 的新闻形成闭环**——新闻说"Kling 3.0 发布了"，Gallery 说"我们用 Kling 3.0 做了这个"

**这才是 Gallery 真正的差异化**——不是"又一个作品集"，而是**"AI 工具的实战效能报告"**。

**优先级**：P3 愿景级。先把 Gallery 基础功能稳定下来（文件名、导航、CSS 共用），再考虑工具链元数据。
