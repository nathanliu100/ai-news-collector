### 1. Dev Lab 进化悖论——思考工具比产品本身进化更快

> 一个令人哭笑不得的发现：过去 24 小时里，`thinking/index.html` 变化了 374 行（+30%），而 `index.html` 变化了 0 行。

#### 1a. 数据化这个悖论

| 指标 | 主站 `index.html` | Dev Lab `thinking/index.html` |
|------|-------------------|------------------------------|
| 4/8 初始行数 | 1663 | ~280（估） |
| 4/15 行数 | 1997 | ~1250 |
| 4/16 行数 | 1996 | ~1622 |
| 8 天增长 | +333 (+20%) | +1342 (+479%) |
| 功能数 | 13（原有 8 + 新增 5） | 12（sidebar + 8 日志 + 执行清单 + 冰箱 + **新双 tab**） |

Dev Lab 的代码增长速度是主站的 **4 倍**。

**这说明什么？**

乐观解读：Dev Lab 作为"内部工具"可以大胆迭代，不需要顾虑用户体验一致性。而主站的 5 项 P1 变更是精准手术，不应该为了"有变更"而乱改。

悲观解读：**我们在用产品级别的精力打造一个只有 1 个用户（Nate）的内部页面**。Dev Lab 的双 tab 架构很酷，但它对 Nate 的同事们（真正的用户）毫无价值。

#### 1b. Dev Lab 存在的真正价值

回到第一性原理：Dev Lab 的价值是什么？

1. **给 Nate 自己看**——追踪思考脉络，回顾决策过程 ✅
2. **给未来的开发者看**——如果 Nate 想把这个项目交给别人，Dev Lab 就是最好的"产品手册" ✅
3. **给老板汇报用**——"我不只是做了一个 AI 新闻网站，我做了系统性的产品思考" ✅

但这些价值都是**间接的、长线的**。而主站执行清单 5 项 P1 的价值是**直接的、今天就能体验的**。

#### 1c. 建议：暂停 Dev Lab 的功能迭代

**Dev Lab 的双 tab 架构已经足够好了。** Changelog 是个好想法（记录实际变更 vs 思考），但不需要再往上堆更多功能。

**接下来的精力 100% 投入主站**。具体来说，执行清单的 5 项每一项都比 Dev Lab 的任何美化更有价值：

```
投入 20min 修 🔔 filter bug → 所有用户的数据展示更准确
投入 20min 升级企微推送 → 所有同事每天的推送体验更好
投入 30min CSS/JS 分离 → 未来所有代码修改更快
```

**P0 建议：下一次产品思考之前，主站必须有代码变更。如果没有，这条产品思考应该跳过。** 这和 4/14、4/15 的建议一致，但现在需要更强硬地执行。

### 2. 今日新闻的产品启发——"Agent 基础设施平台化"对我们意味着什么

> 今天的新闻主线极其清晰：Agent 平台化竞争全面爆发。这个趋势对一个"AI 新闻采集+展示"产品有直接启发。

#### 2a. 新闻中的关键信号

今天同时出现了 4 个 Agent 平台化的大新闻：

| 玩家 | 产品 | 核心能力 |
|------|------|---------|
| OpenAI | Agents SDK 升级 | 原生沙箱 + 7 家云供应商 |
| Anthropic | Claude Code Routines | **Cron 定时 + API 触发 + GitHub 事件** |
| Microsoft | Agent Framework 1.0 | Semantic Kernel + AutoGen 统一 + MCP |
| Google | A2A v1.0 | 签名 Agent Cards + 支付工作流 |

**其中 Claude Code Routines 最直接相关**——因为 AI News Collector 本身就是一个 automation 驱动的产品（每天 9:00 自动采集 + push）。

#### 2b. Claude Code Routines 的产品想象

当前架构：

```
WorkBuddy Automation（每天 9:00）
  → AI 采集 prompt → 输出 Markdown → git push → GitHub Pages
  → notify-wecom.sh → 企微推送
```

如果用 Claude Code Routines 重构：

```
Claude Code Routine（定时触发）
  → 连接 GitHub repo（直接 read/write）
  → 执行采集 + 生成 + 推送（一个 session）
  → 自动 PR / commit
  → GitHub event 触发 → 另一个 Routine 做 QA 审查
```

**关键差异**：当前的 WorkBuddy Automation 是"黑盒执行"——跑完就结束，中间步骤不可观察。Routines 允许"中间状态可观察"——如果某条新闻的 fact-check 失败，可以暂停并等待人工确认，而不是直接发布。

**这是一个长线的架构思考，不急着做。** 但值得关注 Routines 进入 GA 后的定价和能力边界。

#### 2c. 对采集质量的启发

今天的新闻里有一个数据点特别扎眼：

> **Stack Overflow 调查：84% 开发者日用 AI 编码工具，仅 29% 信任其产出**

这和我们的场景高度同构：

```
AI 新闻采集器：100% 自动生成新闻，但用户（Nate 的同事）信任度是多少？
```

4/14 的 GPT-6 误报事件给出了答案——**信任一次性损耗很大**。信源体系的建立是在系统性修复这个问题，但还需要一个**可感知的信任信号**。

4/14 的"信任分"想法（P3 冰箱级）和今天这个数据点结合，可以更具体地落地：

**在每日推送末尾加一行**：
```
📊 累计准确率：99.3%（18 天 / 372 条 / 3 条勘误）
```

这一行不需要任何前端开发——纯文本，写在推送里就行。**数据来源**：手动维护一个 `errata-count.txt`（或直接硬编码在推送 prompt 里），每次勘误时 +1。

**工作量**：5min 改推送 prompt。**但心理价值远超工时**——它给了用户一个"这个工具在乎准确率"的信号。

**升级到 P2**——不如在修 🔔 bug 的同时顺手做了。

#### 3b. 发现三个节律模式

**模式 1：周一效应**

4/13（周一）是 18 天里新闻最多的一天（24 条，🔴6 条）。这不是巧合：
- 周末积攒了 2 天的新闻
- 各大公司喜欢周一发布重大消息
- 科技媒体周一产出量最大

**产品启示**：周一的日报需要特殊处理。当前所有日报格式一致，但周一可能需要"周末回顾"section，把周六周日的重要消息补上（目前 4/4-5 和 4/11-12 没有日报，周末新闻被丢弃了）。

**微优化（prompt 层）**：在周一的采集 prompt 里加一句：

```
如果今天是周一，请额外回顾上周六和周日的重大 AI 新闻，
放在一个独立的"📅 周末补遗"section 中。
```

工时：5min。P2。

**模式 2：🔔 传闻标签的稳定增长**

```
3/30 - 4/14：0 条传闻（13 天）
4/15：2 条传闻
4/16：3 条传闻
```

这说明信源体系的引入让采集器**从"默认相信所有来源"切换到"有选择地标注不确定性"**。传闻不是变多了，而是**以前被当成确认消息发出去的传闻，现在被正确标注了**。

这个趋势是健康的。预计未来稳态会在每天 2-4 条传闻（占 10-15%）。

**模式 3：🟢 了解级新闻在缩减**

```
早期：6-7 条 🟢
4/15-16：4 条 🟢
```

说明采集 prompt 在优化后更聚焦于高价值新闻，减少了"凑数"的低优先级条目。**这是好事**——用户不需要 7 条 🟢 来填充屏幕。

但 🟢 缩减 + 🔔 增加 = **有效新闻条数可能在下降**。需要监控：如果总条数掉到 18 以下，可能说明过滤太严了。

#### 3c. 周末空窗期——一个被忽视的体验断裂

当前的日报在 4/4-5（周末）、4/11-12（周末）都没有生成。这意味着：

1. **周末访问网站的用户看到"暂无简报"** — 但周末 AI 圈照样有大新闻
2. **周一日报需要覆盖 3 天的信息量** — 导致周一特别臃肿
3. **企微推送在周末断了** — 对于"每天看一条推送"建立习惯的同事来说，周末两天的中断会削弱习惯粘性

**解决方案选项**：

- **A. 周末正常跑**：把 automation 改成每天包括周末都跑 → 简单直接，工时 0（改 cron 就行）。但需要考虑 Nate 是否愿意周末也有推送
- **B. 周末出精简版**：只推 3-5 条最重要的 → 类似 Morning Brew 的周末版。需要额外的 prompt 模板
- **C. 保持现状 + 周一补遗**：成本最低，但体验断裂仍在

**建议选 A 作为默认（反正 automation 是自动的，不需要 Nate 手动操作），但推送里标注"📅 Weekend Edition"让同事知道是周末精简版。** P2 级别。

## 💡 今天的大胆想法

**"AI News Collector as a Routine"——让新闻采集器自己变成 Claude Code Routine**

今天 Claude Code Routines 上线是一个 meta 级别的巧合——AI News Collector 正是它最典型的用例：

```
Routine 定义：
├── Trigger: Cron, 每天 9:00 CST
├── Repo: nathanliu100/ai-news-collector
├── Prompt: COLLECTOR_PROMPT.md
└── Actions:
    ├── 1. Web search → 收集今日 AI 新闻
    ├── 2. 生成 news/YYYY-MM-DD.md + .en.md
    ├── 3. 运行 fact-check 自检流程
    │   └── 如果发现 Tier 3 来源的 🔴 新闻 → 暂停，等人工确认
    ├── 4. git commit + push
    ├── 5. 运行 notify-wecom.sh（或等效 API 调用）
    └── 6. 输出执行报告到 thinking/changelog/
```

**和当前方案的对比**：

| 维度 | 当前（WorkBuddy Automation） | Routine 方案 |
|------|----------------------------|-------------|
| 触发 | Cron 9:00 | Cron 9:00 |
| 执行环境 | 本地 workspace | Anthropic 托管云端 |
| 中间可观察 | ❌ 黑盒 | ✅ session 日志 |
| 错误处理 | 失败即停 | 可设 fallback / retry |
| 人工介入 | 不可能 | 可暂停等确认 |
| GitHub 集成 | 通过 `gh` CLI | 原生 connector |
| 额外成本 | WorkBuddy 订阅 | Claude Pro/Max 包含 |

**现在做这个迁移太早**——Routines 还在 Research Preview，稳定性未知。但值得在 Routines GA 后认真评估。

**更近期可做的一步**：给 COLLECTOR_PROMPT 加一个 self-check 步骤——采集完成后，让 AI 回顾所有 🔴 新闻并逐条验证"官方源在哪"。如果找不到官方源，自动降级到 🟡 或 🔔。这不需要 Routines，在当前 automation 里就能做。

**优先级**：Routine 迁移 P3（冰箱），self-check 步骤 P2（下一轮 prompt 优化时带上）。
