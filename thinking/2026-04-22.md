### 1. Gallery 进化——从"展示页"到"影评平台"

> Gallery 在 48 小时内从 843 行增长到 1138 行（+35%）。更重要的是，它的**产品形态发生了质变**。

#### 1a. Gallery 不再是"作品集"了

对比 4/21 和 4/22 的 Gallery：

| 维度 | 4/21（843 行） | 4/22（1138 行） | 变化 |
|------|---------------|----------------|------|
| 作品数 | 2 卡片 | 2 卡片（Hell Grind 替换原有） | 内容更新 |

**关键发现**：Gallery 的 Crew 评论和日报的 Crew 评论是**两种完全不同的东西**：

这不是简单地"把日报的 Crew 搬到 Gallery"——Gallery 的 Crew 评论更接近**专业影评人的圆桌讨论**。6 位评论者从不同角度（商业/产品/科学/内容/影视专业）拆解同一个作品，信息密度极高。

**产品启示**：Gallery 正在从"作品展示页"进化为**"AI 影像评论平台"**。这和主站的"新闻消费"形成了互补——主站告诉你"发生了什么"，Gallery 告诉你"最好的作品是什么 + 为什么好"。

#### 1c. R2 CDN 接入——基础设施的静悄悄升级

Gallery 的视频从本地文件迁移到了 Cloudflare R2：

```
旧：<source src="ai-videos/xxx.mp4">
新：<source src="https://pub-51163ae388a44e2791077f1ad1a427ea.r2.dev/xxx.mp4">
```

**为什么这比看起来更重要**：
1. **性能**：R2 CDN 全球节点，加载速度从"看 GitHub Pages 心情"变成"稳定快速"
2. **容量**：GitHub 仓库不适合存视频文件（大文件 + 仓库膨胀），R2 是对的方案
3. **成本**：R2 的出站流量免费（Cloudflare 特色），比 S3 便宜得多

**这是一个被低估的基建决策**——它意味着 Gallery 未来可以承载大量视频内容，而不用担心仓库大小和加载速度。

### 2. 三个 fix commit 的架构警告

> `c494093` → `1d83b51` → `ba5ab23`——三个连续 fix commit，都在修 DOM 结构和视频源问题。

#### 2a. 问题模式

从 commit message 看：
1. **`c494093`**：restore hero section and filter bar lost during file concat
2. **`1d83b51`**：remove orphaned HTML from file concat, fix DOM structure
3. **`ba5ab23`**：swap video sources to match correct card content

"file concat"这个词出现了两次——说明在编辑 `gallery-mockup.html` 时遇到了**文件拼接/合并导致的结构混乱**。

**根本原因推测**：1138 行的单文件 HTML 在大量编辑（新增 Crew 评论、替换视频源、加 logo 等）过程中，出现了 DOM 元素丢失或错位的问题。当一个 HTML 文件接近 1000+ 行时，人工或 AI 编辑都容易发生"上下文丢失"——改 A 区域时不小心影响了 B 区域。

**这是 CSS/JS 分离需求的第三次升级理由**：

| 时间 | 升级理由 | 优先级 |
|------|---------|--------|
| 4/8 | 单文件太长（1660 行）→ 可维护性 | P2 |
| 4/21 | 多文件间 CSS 重复 → 效率问题 | P0 |
| **4/22** | **文件编辑导致 DOM 结构损坏 → 生产事故风险** | **P0+** |

三个 fix commit 就是活生生的案例——如果 CSS 和 JS 分离出去，`gallery-mockup.html` 的 HTML 骨架会从 1138 行缩减到 ~400 行（纯内容和结构），编辑时出错的概率大幅降低。

#### 2b. 更深层的问题：Gallery 的增长速度

```
4/21: gallery-mockup.html  843 行
4/22: gallery-mockup.html 1138 行  (+295, +35%)
```

如果 Gallery 继续以这个速度增长（每天 1-2 个新作品 + 每个作品 6 人深度评论），一周后就会突破 2000 行。而每个作品卡片 + 6 人 Crew 评论大约占 200-250 行 HTML。

**Gallery 需要一个数据驱动的架构**——作品信息存在 JSON 文件中，HTML 只负责渲染模板。这样新增作品只需要编辑 JSON，不用碰 HTML 结构。

```
当前架构（硬编码）：
  gallery-mockup.html = 模板 + 样式 + 数据
  新增作品 = 修改整个 HTML 文件 → 容易出 DOM 错误

理想架构（数据驱动）：
  gallery.html = 模板 + 渲染逻辑
  gallery.css = 样式（共用 style.css）
  gallery-data.json = 作品数据 + crew 评论
  新增作品 = 在 JSON 中添加一个对象 → 零 DOM 风险
```

**工时估算**：~45min（提取现有 2 个作品为 JSON + 写渲染 JS）。

**优先级**：P2。先做 CSS/JS 分离（P0），再做 Gallery 数据化。但如果 Gallery 继续这种增长速度，这个 P2 可能在一周内升到 P1。

## 💡 今天的大胆想法

**"Crew 角色弹性系统"——不是所有新闻都需要 5 个人评论，也不是所有作品都需要 6 个人评论**

今天的观察触发了这个想法：

- 日报 Crew：5 位，每人 1-2 句快评，适合新闻速览
- Gallery Crew：6 位（含 Tim），每人 80-150 字深度评论，适合影评
- Jobs 在 4/20 圆桌里说过：**"最好的产品功能应该是弹性的，不是刚性的"**

设想一个**角色匹配系统**：

```
新闻类型 → 自动匹配最适合的 Crew 成员

🔴 AI 模型发布：Elon（战略） + 万维钢（数据） + Jobs（产品） = 3 人精评
🔴 设计工具：Jobs（产品） + MrBeast（创作者） + Tim（影视） = 3 人精评
```
