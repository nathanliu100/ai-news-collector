### 1. 从 GPT-6 勘误事件看"信息可信度"——这是一个被低估的产品维度

> 4/13 日报中"GPT-6 定档 4/14 发布"被 4/14 日报修正为"尚未正式发布"。这不是 bug，这是一个**信号**。

#### 1a. 发生了什么

还原时间线：
```
4/13（周一）日报：
  🔴 GPT-6 明日发布 — OpenAI 确认，Symphony 架构，200 万 Token

4/14（周二）日报：
  🟡 GPT-6 预计 4 月内发布 — ⚠️ 尚未正式发布
  → 勘误说明：初版将 GPT-6 误报为"今日正式发布"，
    经交叉验证 OpenAI 官方尚未正式发布该模型
```

从🔴降到🟡，从"确认发布"到"传闻/预期"——这个修正幅度不小。

**为什么会出错？** 大概率是采集 prompt 依赖的新闻源本身不准确。当大量媒体和 X 上 KOL 都在说"GPT-6 定档 4/14"时，AI 采集器很容易把"市场预期"当成"官方确认"。这不是 AI 的问题，而是**信息源的问题**——正好也验证了 MEMORY.md 里刚加的 Fact-check 规范的必要性。

**对用户的影响有多大？**

场景还原：
1. 4/13 下午，同事在企微群收到推送："🔴 GPT-6 明天发布"
2. 同事转发到其他群，甚至给老板汇报"OpenAI 明天要发 GPT-6"
3. 4/14 上午，GPT-6 并没有发布
4. 同事（或老板）回来问："你昨天说的那个 GPT-6 呢？"
5. 信任受损

**这不是假设场景——这是 AI 新闻聚合产品最大的信任风险。**

Morning Brew 的编辑团队是人工核实的；Hacker News 靠社区投票自纠错；我们呢？完全依赖 AI 采集器的判断力。

#### 1b. "信息可信度"作为产品维度

当前我们的新闻有两个维度的标签：
- **重要性**：🔴🟡🟢
- **时效性**：🆕今日 / 🔄持续 / 📌背景

但缺少第三个关键维度：**确信度（Confidence）**

```
当前：
  🔴 GPT-6 明日发布
  （用户理解："确认发布"）

改进后：
  🔴 GPT-6 明日发布 [✅ 官方确认]
  🔴 GPT-6 预计 4 月发布 [⚠️ 多源传闻，尚待官方确认]
  🟡 某模型即将上线 [💬 单一来源爆料]
```

**三级确信度标签**：

| 标签 | 含义 | 判断标准 |
|------|------|---------|
| ✅ 已确认 | 有官方公告/权威媒体确认 | OpenAI blog、TechCrunch 引用官方声明 |
| ⚠️ 高度可信 | 多个独立信源交叉验证 | 3+ 家媒体报道，但无官方声明 |
| 💬 传闻/预期 | 单一来源或社交媒体爆料 | X 上 KOL 发的、一篇 CSDN 博客 |

**这个改动的价值**：

1. **对用户**：一眼知道该信息的可靠程度，不会被"AI 误报的假确认"坑到
2. **对产品**：把"偶尔出错"变成"系统性标注不确定性"——错误不可避免，但诚实标注可以维护信任
3. **对采集质量**：强制 AI 在生成时思考"这条新闻的来源有多可靠"，间接提升采集质量
4. **对 Nate 的快报**：周报引用日报内容时，可以只引用 ✅ 确认的消息，避免在正式快报里传播传闻

**工作量评估**：
- prompt 改动：在采集 prompt 里增加确信度标签规则 → 30min
- 前端展示：如果确信度标签混在正文里（Markdown 格式），不需要改前端。但如果想做成可视化的 badge（比如绿色/黄色/灰色小标签），需要额外的 CSS + JS → 1h
- **建议先做 prompt 层面的标注（P1），前端视觉化放在 V2**

### 2. 勘误机制：从"偷偷改"到"公开透明"

> 4/14 日报顶部有一行 `⚠️ 勘误说明`。这很好——但可以更系统化。

#### 2a. 当前的勘误处理

4/14 日报的做法：
```markdown
> ⚠️ **勘误说明**：初版将 GPT-6（代号 Spud）误报为"今日正式发布"，
> 经交叉验证 OpenAI 官方尚未正式发布该模型，已更正为预期/传闻。
```

**做得好的**：
- ✅ 主动承认错误，没有偷偷把 4/13 的内容改了
- ✅ 在引言区域明确标注
- ✅ 具体说明了什么被改了

**可以更好的**：
- ❌ 4/13 的原日报没有被更新或标注"此版本已有勘误"
- ❌ 如果用户只看 4/13 的页面（比如企微推送链接过去的），看到的还是旧版本
- ❌ 没有一个系统化的勘误记录——如果半年后回看，不知道哪些新闻曾被纠错

#### 2b. 勘误系统的设计思考

**Level 1：最小可行的勘误机制（纯 Markdown）**

在被勘误的日报（如 4/13.md）中追加一个 section：

```markdown
---

## ⚠️ 勘误记录

| 更新日期 | 原始内容 | 更正内容 | 原因 |
|---------|---------|---------|------|
| 4/14 | GPT-6 定档 4/14 发布 [🔴] | GPT-6 预计 4 月内发布 [🟡] | 经验证 OpenAI 尚未正式发布 |
```

这样回看 4/13 时能看到"这条信息后来被纠正了"。

**Level 2：前端勘误标识**

在 `index.html` 里检测 Markdown 中是否包含 `## ⚠️ 勘误记录`，如果有：
- 在 stats ribbon 上显示一个小的 "⚠️ 1 errata" 标记
- 被勘误的那条新闻加一个删除线 + 旁注

**Level 3：跨日勘误关联**

如果 4/14 纠正了 4/13 的内容，理想状态下：
- 4/13 页面显示："此日报中关于 GPT-6 的内容已在 4/14 勘误"
- 4/14 的勘误说明链接到 4/13 的原文位置

但 Level 3 涉及跨文件链接，在纯静态方案下实现复杂度太高。**先做 Level 1 就足够了。**

#### 2c. 对采集 prompt 的启发

勘误的根源是采集质量。与其亡羊补牢，不如从源头提高准确率。

**4/14 的 fact-check 规范**（已写入 MEMORY.md）是正确的方向，但可以更具体地落到 prompt 里：

```
采集新闻时，对以下类型的信息必须标注确认状态：
1. 模型发布/产品上线：必须找到官方 blog 或官方 X 发文。
   如果只有媒体转述，标注为"据[XX]报道"而非"官方确认"
2. 融资/收购：必须找到 TechCrunch、CNBC、Bloomberg 等权威商业媒体
   的独立报道。自媒体的"爆料"不算
3. 技术性能指标：必须有论文、官方 benchmark 或可复现的第三方评测。
   "据说性能提升 40%"不等于"性能提升 40%"
4. 预测/传闻：必须标注信息来源的类型和数量。
   例如"据 3 家媒体报道"vs"据 X 用户爆料"
```

**这条 prompt 改动应该加入 P0 执行清单，和之前的"弹性条数 + 超级新闻日"合并执行。**

## 💡 今天的大胆想法

**"信任分"——给 AI News Collector 一个累计信誉度量**

设想在网站某处（也许是 footer 或 About 页面）显示一个"信任指标"：

```
📊 采集准确率
━━━━━━━━━━━━━━ 96.2%
过去 30 天 | 308 条新闻 | 12 条勘误
```

这不是为了炫耀——而是**自我约束**。

公开展示准确率意味着：
1. 每次出错都有代价（指标下降），倒逼采集质量持续提升
2. 用户看到这个数字会产生信任感——"他们至少在 tracking 自己的准确率"
3. 如果某天准确率骤降，说明采集流程出了系统性问题，需要排查

**计算方式**：
```
准确率 = (总新闻条数 - 被勘误条数) / 总新闻条数 × 100%
```

**落地路径**：
- 短期：先手动维护一个 `errata.json`（记录每次勘误的日期、原文、更正）
- 中期：在前端从 `errata.json` 读取并计算展示
- 长期：在采集 automation 里加一个 self-review 步骤（AI 对比前一天的预测 vs 今天的事实，自动识别需要勘误的条目）

**参考**：Nate Silver 的 FiveThirtyEight 网站长期公开显示预测准确率，这个透明度成了他们最大的竞争壁垒之一。AI 新闻采集器如果也这么做，在同类产品中可能是独一无二的。

**优先级**：P3（这是 V2 以后的事，但值得现在就开始收集数据）。
