## 三、"超级新闻日"的用户体验审视

### 3a. 今天为什么是"超级日"

4/29 日报的信息密度比日常高出 50%：

```
日常水平：~18-20 条新闻，🔴 1-2 条
4/29 水平：22 条新闻，🔴 4 条（翻倍）
```

而且 4 条 🔴 全部是重量级：
1. Musk 出庭作证（世纪审判进入新阶段）
2. OpenAI × Microsoft "分手"（AGI 条款废除）
3. NVIDIA Nemotron 3 Nano Omni（开源全模态）
4. 白宫约谈 Claude Mythos（国家安全级别）

**今天的 4 条 🔴 随便拿一条出来，放在其他日子都能单独做头条。** 四条堆在一起，信息过载的感觉非常明显。

### 3b. 信息过载的具体体验问题

**问题 1：今日必看区的"决策疲劳"**

用户看到 4 条 🔴 都带"必看"标记，但注意力只够消化 1-2 条。他们会：
- 选择最吸引自己的那一条深入看（通常是标题最戏剧化的）
- 对其余 3 条产生"已知但没读"的认知负担

**数据验证点**：明天看 Analytics，如果超级新闻日（4/24 GPT-5.5 + 4/28 审判日）的 UV 显著高于普通日但停留时间没有成比例增加，说明"更多头条 ≠ 更深阅读"。

**问题 2：编辑观察的"三重交汇"太长了**

今天的编辑观察有 3 条叙事线 × 每条 3-5 行分析 = ~15 行纯文本。放在页面底部，用户可能已经精力耗尽了。

**优化方向（纯 prompt 层面）**：
- 超级新闻日（🔴 ≥ 4）时，编辑观察精简为一段话 + 一句判断
- 或者反过来——超级日的编辑观察更激进，只给一个"一句话总结"：`当一项技术同时需要法庭审判、政府闭门会谈和国际条约来管理时——它是文明级议题。` 这一句比三段分析更有力

### 3c. 对"今日必看 5 条上限"的回顾

Day 23（4/23 晚间版）提出过"今日必看上限 5 条"机制。今天 4 条刚好在 5 以内。但 4 条全是世界级新闻，读者能消化吗？

**更精确的思考**：问题不是"数量"而是"权重分配"。

```
当前："今日必看"4 条全部是平等列表
理想："今日必看"有主次之分

方案（纯格式优化）：
  ⚡ 今日必看
  
  📌 头号关注：OpenAI × Microsoft 正式"分手"——AGI 条款废除
  
  · Musk 出庭作证：世纪审判第一天
  · NVIDIA 发布全模态开源模型 Nemotron 3
  · 白宫紧急约谈：Claude Mythos 引发国安焦虑
```

**把 4 条 🔴 分为"头号关注" × 1 + "同日重要" × 3**。让读者的注意力有一个着力点，而不是面对 4 条"同等重大"的新闻不知道从哪开始。

**实现方式**：纯 COLLECTOR_PROMPT 规则修改。让 AI 采集器在 🔴 ≥ 3 时选出"今日份量最重的一条"加 📌 标记。零代码。

**优先级**：P3。属于"锦上添花"的内容策略优化，不紧急。

## 五、一个从未思考过的维度：首次加载速度

### 5a. 为什么今天要想这个

今天在审查 `index.html` 时注意到一个事实：

```html
<script defer src="https://cdn.jsdelivr.net/npm/marked/marked.min.js"></script>
<link href="https://fonts.googleapis.com/css2?family=Playfair+Display:wght@400;600;700;900&family=Noto+Sans+SC:wght@300;400;500;600;700&family=Inter:wght@300;400;500;600;700&family=JetBrains+Mono:wght@400;500&display=swap" rel="stylesheet">
<link rel="stylesheet" href="assets/style.css">
<script defer src='https://static.cloudflareinsights.com/beacon.min.js' ...></script>
```

**外部依赖清单**：
1. marked.js（CDN → jsdelivr）
2. Google Fonts（4 个字族 × 多 weight）
3. style.css（本地）
4. Cloudflare Analytics beacon
5. app.js（本地）

**核心瓶颈**：Google Fonts。

`Playfair Display`（4 weights）+ `Noto Sans SC`（5 weights）+ `Inter`（5 weights）+ `JetBrains Mono`（2 weights）= **16 个 font weight 文件**。

在国内访问 `fonts.googleapis.com` 的速度不稳定——有时候 2 秒，有时候 10 秒，有时候超时。**如果字体加载慢，页面会 FOUT（Flash of Unstyled Text）或者等很久才渲染完成。**

### 5b. 量化影响

```
加载顺序（理想情况）：
  DNS + HTML:           ~100ms
  style.css:            ~50ms（本地 GitHub Pages）
  Google Fonts CSS:     ~200-1000ms（国内不稳定）
  Fonts 文件下载:       ~500-3000ms（16 个 weight）
  marked.js:            ~100-500ms（jsdelivr，国内通常OK）
  app.js:               ~50ms（本地）
  fetch(news/MD):       ~100-500ms（GitHub Pages）
  marked.parse():       ~50ms
  structureNewsBlocks:  ~20ms
  
  最快情况：~1.5 秒首屏可用
  国内最坏情况：~5-8 秒（字体卡住时）
```

### 5c. 优化方向

1. **`font-display: swap`**——Google Fonts URL 已经默认带 `&display=swap`（检查：URL 中是否有 `display=swap` 参数）
   - ✅ 实际 URL 已包含 `display=swap`——字体不会阻塞渲染，会先用系统字体再替换
   
2. **减少 font weights**——16 个 weight 太多了
   - Playfair Display: 真的需要 4 个 weight（400/600/700/900）？标题用 700 或 900 一个就够
   - Noto Sans SC: 5 个 weight（300/400/500/600/700），正文 400 + 标题 600 = 2 个够了
   - Inter: 5 个 weight，同理削减到 400 + 600
   - JetBrains Mono: 2 个 weight → 保留 400 一个
   - **从 16 个 → 6-7 个 weight**，文件大小和请求数减半
   
3. **Noto Sans SC 是大头**——CJK 字体每个 weight 可达 2-5MB
   - 考虑只加载 `Noto Sans SC:wght@400;700` 两个
   - 或者用 `unicode-range` 子集化（只加载常用字）
   - 或者直接用系统中文字体 fallback（`-apple-system, 'PingFang SC', 'Microsoft YaHei'`）

**工时估算**：10 分钟（修改 Google Fonts link URL + CSS fallback 调整）。
**优先级**：P3。目前网站主要在部门内使用，用户容忍度高。但如果 Analytics 显示国内用户跳出率高，这可能是原因之一。

## 六、"31 天问题"——3/30 日报今天正式过期了

### 6a. 发生了什么

`scanFiles()` 扫描最近 30 天的历史。今天 4/29 - 30 天 = 3/30。

**3/30 是 AI News Collector 的第一份日报。从今天起，它从 History Panel 中消失了。**

这意味着：
- 网站的"创始日"内容对新用户不可见
- 最早 3 天的日报（3/30、3/31、4/1）将陆续消失
- 每天都会有一份日报"过期"

### 6b. news/index.json 的紧迫性上升

Day 20 提出的 `news/index.json` 方案现在不再是"远期优化"——它是一个**正在发生的功能退化**。

```
当前情况：
  3/30  已过期 ← 今天
  3/31  明天过期
  4/01  后天过期
  ...
  每天丢失 1 份历史
```

**但考虑到实际用户行为**：有多少人会翻看 30 天前的日报？大概率非常少。这个问题的"感知严重性"远低于"实际严重性"。

**明天 Analytics 可以验证**：如果 Top Paths 中没有任何 `news/2026-03-*` 的访问记录，说明没人翻旧日报——index.json 优先级可以继续保持 P2。

### 6c. 最小应急方案

如果不想现在做 index.json，有一个 10 秒 fix：把 `scanFiles()` 的 30 天上限改成 60 天。

```javascript
// 现在：30 天
for (var i = 0; i < 30; i++) { ... }

// 临时修：60 天
for (var i = 0; i < 60; i++) { ... }
```

**代价**：60 次 HEAD 请求。对用户体验有微小影响（History Panel 打开时稍慢），但不会造成功能问题。

**这不是最终方案**（index.json 才是），但能争取 30 天缓冲期。

**优先级**：P3（如果月底 index.json 还没做，这个 1 行改动可以先顶上）。
