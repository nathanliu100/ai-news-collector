## 🔍 今天的思考

### 1. 信息密度 vs 可读性的平衡

**现状**：每天 20+ 条新闻全部平铺在一个长页面里，每条都是 H3 + 正文的结构。用户要么看「今日必看」5 条就走，要么得滚很长才能看完。

**痛点**：对于"快速扫一眼"的用户（可能是 Nate 转发到企微群后点进来的同事），22 条全展开太重了。但对于深度读者，又确实需要完整信息。

**可能的方向**：
- **折叠式交互**：分类标题点击展开/收起，默认只展开「今日必看」和用户点击的分类。这样首屏信息密度高，深度内容按需展开
- **紧凑模式 / 展开模式** 切换：类似 Hacker News 的列表模式 vs 详情模式
- **阅读进度**：右侧小型 TOC（目录导航），让用户知道自己在哪个位置，快速跳转

### 2. 「今日必看」的视觉权重不够

**现状**：`⚡ 今日必看` 是一个 H2 标题 + 有序列表，视觉上和其他分类没有明显区分。它本应是整个页面的 **Hero Section**。

**优化想法**：
- 给「今日必看」一个独特的卡片式设计：大字号、背景色块、左侧红色竖线、数字编号更醒目
- 参考 The Verge 或 Morning Brew 的 "Top Stories" 设计——让它在视觉上一眼就知道是最重要的
- 或者做成一个横向滑动的卡片组（mobile 端尤其好用）

### 3. 移动端体验

**现状**：有完整的 responsive 设计，移动端 filter bar 在底部，还算不错。

**但**：
- 底部 filter bar 只有 All/🔴/🟡/🟢 四个按钮，**但用户第一次来根本不知道这些圆圈代表什么**。缺少 onboarding/tooltip
- 移动端 nav-date 区域比较拥挤，日期 + 前后箭头 + 右侧多个按钮挤在 48px 高度里
- 长内容页面没有 **"回到顶部"** 按钮——滚到底部之后要手动滑很久

### 4. 企微推送 → 网页的衔接

**场景**：同事在企微群里看到推送摘要，觉得某条新闻有意思，点链接进来看完整版。

**问题**：点进来后直接到最新日期的页面，但推送里提到的具体新闻条目**没有锚点定位**。用户得自己去找。

**优化**：
- 给每条新闻（H3）生成 anchor ID
- 企微推送的链接带上 `#anchor`，让用户点进来直接跳到对应新闻
- 甚至可以：点击推送中的具体条目 → 直接高亮该条新闻

### 5. 搜索功能 = 缺失的大件

**现状**：完全没有搜索。如果同事问"上周那个 DeepSeek V4 的新闻在哪？"，目前只能手动翻日期。

**用户痛点**：随着日报积累（已经 11 天了，很快会到 30+ 天），找历史信息会越来越痛。

**方案**：
- 全文搜索：基于已加载的 Markdown 做 client-side 搜索（数据量不大，30 天约 400KB MD，纯前端搜完全可行）
- 搜索结果高亮 + 日期标注
- 可以放在 Archive 面板里，或者作为独立搜索栏

## 🔍 晚间深度思考（18:00 更新）

> 下午重新审视网站，换一个角度：不看"缺什么功能"，而是看"现有功能是否在正确地工作"。

### 6. 单文件架构的技术债务正在累积

**现状**：`index.html` 已经 1660 行，CSS、JS、HTML 全部耦合在一起。

**问题清单**：
- **CSS 变量命名**已经有 30+ 个，但没有 design token 体系，`--surface` / `--surface-alt` / `--bg` 三者的关系只能靠注释猜
- **JS 全局变量**满天飞：`currentDate`, `currentFilter`, `currentMarkdown`, `currentMode`, `weeklyDate` 都是裸露的全局状态。一旦功能增多（比如加搜索），状态管理会变成噩梦
- **没有 build 工具**：直接裸 HTML 没问题，但未来加图片压缩、CSS minify、代码拆分都得从零搭
- **marked.js 从 CDN 加载**：万一 CDN 挂了（jsdelivr 在国内偶尔抽风），整个网站就白屏

**不是说现在要重构**，但值得记录一下"技术债拐点"：大概在 **2000 行**左右，单文件维护成本会急剧上升。现在 1660 行，距离拐点不远了。

**轻量化方案（不引入构建工具）**：
- CSS 抽出去变成 `style.css`
- JS 抽出去变成 `app.js`
- 本地备一份 `marked.min.js`（fallback）
- 这样至少可以让 `index.html` 只剩骨架，三个文件各司其职

### 7. 数据结构的隐性问题：新闻去重

**场景**：今天（4/8）和昨天（4/7）有不少重复的新闻。比如：
- "Q1 全球创投 2970 亿"在 4/6、4/7、4/8 连续三天出现
- "可灵 3.0" 相关新闻在 4/6、4/7、4/8 各有不同角度的报道
- "AI 视频格局重塑" 主题几乎天天有

**用户感知**：如果一个同事连续三天打开网站，他会觉得"好像每天都在说同一件事"——虽然内容确实有增量，但没有明确标识。

**优化方向**：
- **续报标签**：对跨日追踪的新闻，标注 `📡 续报第 N 天` 或 `🔄 持续跟踪`（其实已有 🔄 标签，但使用不够系统）
- **首报 vs 续报分流**：首次出现的新闻权重更高，续报可以用更紧凑的展示（一句话摘要 + "详见 4/6 首报"链接）
- **新闻聚合线程**：同一事件跨多天的报道汇聚成一个"故事线"——这是 Apple News、Google News 的核心体验

### 8. 「编辑观察」的价值被埋没了

**现状**：每天日报最底部有一段 `🔭 编辑观察`，是整份日报中**最有价值的内容**——它不是搬运新闻，而是编辑视角的趋势解读和部门关联分析。

**问题**：
- 它在页面最底部，90% 的用户可能滚不到那里就离开了
- 在企微推送中只有「今日必看」，编辑观察完全不可见
- 它和「📊 数据快照」挨在一起，但两者性质完全不同（数据 vs 观点），视觉上没有区分

**优化想法**：
- **位置提升**：把「编辑观察」放到「今日必看」正下方，作为整份日报的第二大模块。先给结论，再展开细节
- **视觉差异化**：用不同的排版风格（比如全宽背景色块 + 更大的行距 + 手写体引号），让它看起来像杂志的"主编寄语"而非新闻条目
- **企微推送增强**：除了「今日必看」列表，额外加一段编辑观察的核心句（一两行就行），提升推送的"洞察感"

### 9. 周报模式的内容发现问题

**现状**：目前只有 1 份周报（W14）。在"Weekly"模式下，用户按箭头翻周，大部分周是空的。

**体验问题**：
- 用户不知道哪些周有周报、哪些没有，只能盲翻
- 空周的 empty state 写着"该周的周报尚未发布"，但没有提供回到有内容的周的快捷方式
- 没有"最新周报"的明确入口——用户得猜这周有没有

**优化**：
- 周报模式的 Archive 面板应该只显示有周报的周，而不是连续 30 天
- 或者更简单：**取消独立的周模式导航**，改为在 Archive 面板里按类型分组显示（日报列表 + 周报列表），点击即可跳转
- 在日报页面底部加一个"📰 本周周报已发布"的横幅提醒（如果当前周有周报）

## 💡 一个更大的想法

**新闻故事线（Story Thread）**

不再把新闻当作"每天独立的列表"，而是构建跨天的故事线：

```
📖 DeepSeek V4 故事线
├── 4/3  首报：V4 确定 4 月下旬发布，梁文锋亲自带队
├── 4/7  续报：V4 与腾讯混元同月竞争
└── 4/8  最新：V4 "条件记忆"机制细节曝光

📖 AI 视频格局 故事线  
├── 4/6  首报：Sora 关停，可灵 3.0 领跑
├── 4/7  续报：两年巨变复盘，Seedance 2.0 对比测评
└── 4/8  最新：可灵 3.0 Omni 架构升级
```

这种"新闻线程"视图需要：
1. 每条新闻在 Markdown 里加一个 `<!-- thread: deepseek-v4 -->` 之类的隐藏标记
2. 前端解析标记，聚合成故事线
3. 用户可以选择"按日期"或"按故事线"两种浏览模式

工作量大，但**这是和普通新闻聚合器拉开差距的关键特性**。Morning Brew 做不到这个（它是 newsletter），Hacker News 做不到这个（它是社区投票），但一个由 AI 每天系统整理的新闻站可以。
