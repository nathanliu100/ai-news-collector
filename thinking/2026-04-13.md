### 1. 超级新闻日压力测试：明天 GPT-6 发布，网站准备好了吗？

> GPT-6 定档 4/14。按照当前的采集逻辑，明天的新闻日将会是 AI News Collector 上线以来信息密度最高的一天。这个事件暴露了产品在"极端场景"下的哪些结构性问题？

#### 1a. 内容量暴增 vs 固定条数天花板

当前采集器每天稳定输出 21-24 条新闻。但明天可能发生：
- GPT-6 本身就能拆出 5-8 条独立新闻（功能发布、技术架构、定价、竞品反应、用户评测、中国区可用性、对 Claude/Gemini 的影响...）
- 同时 AI 行业不会因为 GPT-6 停转——其他新闻照常产生
- 保守估计明天值得报道的新闻有 35-40 条

**用 22 条的框框装 40 条的信息量——要么砍得血肉模糊，要么重要的东西被挤掉。**

这在 4/12 的"度量审计"里已经提过弹性条数机制，但今天可以更具体地设计：

**"超级新闻日"模式**：
```
普通日：15-22 条（当前模式）
重大日：25-30 条（当日有 2+ 条🔴级新闻涉及同一事件时自动触发）
超级日：30-40 条（年度级事件，如 GPT-6 发布、行业重大监管政策）
```

触发判断不需要多复杂——在采集 prompt 里加一句：

> 如果今天某个单一事件产生了 3 条以上独立新闻，则自动将总条数上限放宽至 30 条，并在引言区域标注"📌 今日为重大事件日，扩展收录"。

**前端影响**：条数增加意味着页面更长。在当前的长列表设计下，从 22 条增加到 35 条，移动端需要多滚动约 60%。这强化了"折叠/展开"交互的需求——但这是 V2 特性，暂不做。

#### 1b. "今日必看"在超级日的表现

当前每天固定 4-5 条"今日必看"。明天的情况：
- GPT-6 可能独占 2-3 条必看位
- Claude Mythos 和 GLM-5.1 的后续反应也可能有新进展
- 其他领域（AI+设计、国内 AI）的重要新闻会被彻底淹没

**问题**："今日必看"的筛选机制没有"同一话题去重"的概念。明天可能出现 5 条必看里有 3 条都是 GPT-6 相关——对已经知道 GPT-6 重要的用户来说，信息冗余。

**优化建议**（prompt 层面）：
```
今日必看选择规则：
- 每个独立事件最多占 2 个必看位
- 优先保证领域多样性：模型 × 工具 × 市场 × 设计各占至少 1 席
- 如果当天有超级事件（如 GPT-6），用 1 条综合性必看替代 3 条拆分
  例如："GPT-6 全球发布：200万Token、5万亿参数、Agent引擎——一文总览"
```

#### 1c. 超级日的排版策略

当前所有新闻视觉权重一致。在超级日里，这个问题会被放大——用户打开网站，看到密密麻麻 30+ 条新闻，**信息过载感翻倍**。

**一个不需要改代码的临时方案**：

在采集 prompt 里调整 Markdown 输出格式——超级新闻日在「今日必看」后面加一段「🗞️ 超级事件特报」section，用 H2 标题独立成块：

```markdown
## 🗞️ 超级事件特报：GPT-6 全球发布

### 🔴 GPT-6 发布总览：200万Token + Symphony全模态架构
...

### 🔴 GPT-6 定价与可用性：Pro/Plus/Team 分级
...

### 🟡 GPT-6 vs Claude Opus 4.6 vs Gemini 3.1 Pro 首轮对比
...

---

## 🧪 其他模型 & 更新（非 GPT-6 相关）
...
```

这样信息架构变成：**特报（深度覆盖一个事件）→ 常规分类（覆盖其他一切）**。用户可以选择只看特报，也可以继续往下滚看全景。

**工作量**：0 前端代码。纯 prompt 改动。但这个改动的时效性很强——**今天就需要决定是否在明天的采集中启用**。

### 2. 新闻消费心理学：22 条新闻，用户真的在"读"吗？

> 5 天的产品思考一直围绕"怎么让网站更好"，但从没问过一个根本性问题：用户消费新闻的真实行为是什么？

#### 2a. 新闻消费的三种模式

参考 Nielsen Norman Group 的研究和 Morning Brew 的用户行为数据，新闻消费大致分三种模式：

| 模式 | 行为 | 时长 | 占比（推测） |
|------|------|------|-------------|
| **扫标题（Scanner）** | 只看 H3 标题 + emoji，不读正文 | 30-60 秒 | 60% |
| **选读（Picker）** | 先扫标题，点击 2-3 条感兴趣的展开读 | 2-3 分钟 | 30% |
| **通读（Reader）** | 从头到尾读完，包括编辑观察 | 5-8 分钟 | 10% |

**如果 60% 的用户是 Scanner，那当前设计对他们的体验优化了多少？**

答案是：**几乎为零**。

Scanner 的需求是：打开 → 30 秒扫完今天有什么 → 关掉。他们需要的是：
- ⚡ 今日必看足够醒目（✅ 目前有，但视觉层级不够突出）
- 标题自解释（✅ 当前标题质量不错）
- **能一眼看到总共多少条、重大几条**（✅ stats ribbon 做到了）
- **不需要滚动就能获取核心信息**（❌ 必看区域在折叠线以下）

#### 2b. 首屏问题："折叠线"以上用户能看到什么？

在桌面端（1440×900 视口），打开网站后首屏能看到：
```
┌────────────────────────────────────────┐
│          AI Daily Briefing             │  ← Masthead
│    Intelligence · Design · Trends      │
├────────────────────────────────────────┤
│  ← 2026-04-13 Mon  All Critical ...   │  ← Nav bar
├────────────────────────────────────────┤
│  Daily | Weekly                        │  ← Mode tabs
├────────────────────────────────────────┤
│  Total 24 | 🔴 6 | 🟡 12 | 🟢 6 ...  │  ← Stats ribbon
├────────────────────────────────────────┤
│                                        │
│  # 🗞️ AI 每日新闻速递 - 2026-04-13    │  ← H1 标题
│                                        │
│  > 今日共收集 24 条新闻...             │  ← 引言 blockquote
│                                        │
│  ───────────────────                   │  ← HR
│                                        │
│  ## ⚡ 今日必看                        │  ← 刚刚出现
│  （列表可能被截断）                     │
└────────────────────────────────────────┘
```

Masthead + Nav + Tabs + Stats + H1 + 引言 + HR = 大约占了 **380px** 的纵向空间。在 900px 视口下，首屏内容区只剩 ~520px。如果 H1 和引言较长，「今日必看」的列表可能只露出前 1-2 条。

**移动端更严重**：iPhone 15 视口高度 844px，但考虑浏览器 URL 栏，实际可视区约 740px。同样的 masthead/nav/tabs/stats 结构在移动端大约占 280px，留给内容区约 460px——但移动端的 H1、引言、字号都经过缩放，「今日必看」大概能完整展示。

**核心问题**：首屏的"装饰性元素"（masthead 的标题和副标题、mode tabs）占了过多纵向空间，挤压了内容。

#### 2c. 针对 Scanner 的低成本优化

不大改架构，只做三处微调：

**① Masthead 滚动折叠**：用户向下滚动时，masthead 自动收起（用 IntersectionObserver），让 nav bar 成为视觉顶部。已有 `nav-bar { position: sticky; top: 0; }` 的基础设施，masthead 只需要在滚动后隐藏或缩小。

这不是新想法（很多新闻网站这么做），但效果显著——**节省 60-80px 的首屏空间**。

```css
/* 概念代码，不实际执行 */
.masthead.collapsed {
    padding: 8px 0 6px;
    border-bottom-width: 1px;
}
.masthead.collapsed .masthead-title { font-size: 16px; }
.masthead.collapsed .masthead-sub { display: none; }
```

**② Stats ribbon 合并进 nav bar**：当前 stats ribbon 独占一行。如果把"24 条 | 🔴6 🟡12 🟢6"放到 nav bar 的右侧（替换 filter pills 的位置），可以**省掉一整行（~50px）**。Filter pills 可以移到 stats ribbon 的弹出 dropdown 里。

但这会增加 nav bar 的拥挤度。需要权衡。**可能更适合桌面端，移动端保持现状。**

**③ "今日必看"固定在首屏**：最激进的方案——把「今日必看」从 Markdown 渲染流中拆出来，变成一个独立的 UI 组件，固定在 stats ribbon 下方、正文内容上方。这样无论怎么滚动，必看区域始终可见（或在滚动 1 屏后折叠为一行摘要）。

```
思考评估：方案③ 的改动量太大（需要 JS 解析 Markdown 并拆分渲染），
ROI 不高。暂时不做，但记录为 V2 备选。
```

**最终建议**：方案①（masthead 滚动折叠）ROI 最高，20 行 JS + 5 行 CSS 就能搞定，首屏多出 60px = 用户更快看到今日必看。加入执行清单。

### 3. 两周运营回顾：产品的"真实用户价值"在哪里？

> 网站跑了 15 天。如果今天下线这个网站，会有人发现吗？会有人抱怨吗？

这是一个残酷但必要的问题。

#### 3a. 当前的"不可替代价值"盘点

| 功能 | 是否有替代品 | 我们的差异化 |
|------|-------------|-------------|
| AI 新闻聚合 | ✅ Google News、Feedly、即刻 | 面向 IEG 游戏设计团队定制的 8 分类 + 💎 标签 |
| 每日自动采集 | ✅ 任何 RSS + AI 摘要工具 | 零人工介入，内容质量一致 |
| 企微推送 | ❌ 没有直接替代 | **这是真正的差异化**——直接推到工作场景 |
| 中英双语 | 部分替代（Google Translate） | 原生双语，非翻译质感 |
| 周报 | ✅ 手写周报 | 基于日报自动聚合，省时间 |
| 编辑观察 | ❌ AI 生成的有态度观点 | 不是冷冰冰的摘要，而是有叙事的分析 |

**最大的不可替代价值是"企微推送"**——它让信息找到人，而不是人去找信息。网站本身更像是推送的"落地页"。

**这意味着什么？**

**企微推送的体验权重应该 ≥ 网站浏览体验**。但过去 5 天的产品思考 80% 都在想网站怎么改、CSS 怎么调——只有 4/11 的锚点方案涉及了企微推送。

**重新校准优先级**：

| 优先级 | 之前的判断 | 校准后 |
|--------|----------|--------|
| 新闻锚点 | P0 | **P0 — 确认，核心在于企微→网站的衔接** |
| 企微推送格式优化 | 没独立列出 | **P0 — 应该和锚点同优先级** |
| OG Meta | P0 | P1 — 重要但不如推送体验紧急 |
| 排版四件套 | P1 | P1 — 有价值但不是核心 |
| 采集 prompt 优化 | P1 | **P0 — 超级新闻日场景让这个变得紧急** |

#### 3b. 企微推送的优化深度

当前推送格式（推测）：
```
⚡ AI 新闻速递 2026-04-13

🔴 GPT-6 定档 4 月 14 日
🔴 Anthropic 发布 Claude Mythos Preview
🔴 智谱 GLM-5.1 MIT 开源
🔴 Sam Altman 住宅遭燃烧弹袭击

👉 https://nathanliu100.github.io/ai-news-collector/
```

**问题 1：缺少"为什么我应该在意"的一句话**

同事收到推送，看到"GPT-6 定档 4/14"——然后呢？这条和我的工作有什么关系？**缺少一个 1 秒钟就能回答 "So What" 的 hook**。

改进后：
```
⚡ AI 新闻速递 2026-04-13

🔴 GPT-6 明天发布 — 200万Token + Agent引擎，可能改变我们的工作流
🔴 Claude Mythos Preview — Opus之上的超级模型，仅50家企业可用
🔴 GLM-5.1 开源 — MIT协议，8小时持续工作，SWE-Bench 全球第一
🔴 Sam Altman 住宅遭燃烧弹袭击 — AI行业领袖安全问题升级

📰 完整日报(24条) → [链接]
```

每条加了一个破折号后的"一句话 So What"。

**问题 2：缺少"今天的一句话总结"**

在链接之前加一行编辑推荐：

```
💬 今日一句话：三大阵营（OpenAI/Anthropic/智谱）在48小时内各亮王牌，
AI的"三国演义"进入决战阶段。
```

**这一行是整条推送的灵魂**——它给了用户一个"叙事框架"来理解今天的 5 条新闻。没有它，5 条新闻就是 5 个孤立的事件。

**工作量**：纯 prompt 改动。在企微推送的生成 prompt 里加两条规则：
1. 每条必看后面加破折号 + "So What"一句话
2. 在链接前加"今日一句话"编辑推荐

## 💡 今天的大胆想法

**"GPT-6 发布日特刊"——做一次事件驱动的内容实验**

明天 GPT-6 全球发布。与其用标准的每日模板来报道，**不如做一期"特刊"**：

**格式设想**：
- 不走常规 8 分类结构，而是以 GPT-6 为中心展开
- 结构：`发布概要 → 技术深度 → 竞品影响 → 部门落地 → 用户评测（如果有） → 编辑长评`
- 在 nav bar 或 masthead 上加一个醒目的标识："📌 Special Edition"
- 如果明天同时有非 GPT-6 的重要新闻，在特刊末尾附一个"其他新闻简讯"section

**为什么值得做**：
1. 测试"事件驱动内容"的用户反应——如果特刊的企微点击率明显高于日常推送，说明这种格式有价值
2. 让网站有"呼吸感"——不是每天机械重复同一格式，偶尔来一期特刊，用户会期待"下一次特刊是什么时候"
3. GPT-6 是完美的实验素材——它足够大、足够有深度、足够有话题性

**风险**：
- 需要临时调整明天的采集 prompt，时间紧
- 如果 GPT-6 实际发布内容不如预期（延期/缩水），特刊会显得小题大做

**决定**：如果明天 GPT-6 确实按计划发布了，用特刊格式。如果延期或只是小更新，回退到标准格式。**需要在明天 9:00 采集前确认。**
