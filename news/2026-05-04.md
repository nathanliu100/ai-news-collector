### 🔴 [VS Code 强制插入 "Co-Authored-by Copilot" 署名引发开发者社区地震](https://www.thestackstories.com/blog/vs-code-copilot-auto-commits)
**来源**：Hacker News / VS Code GitHub PR #310226 [🏢 官方] [🟩 T1] | **时间**：2026-05-03 | ✅ 已验证：VS Code 官方 GitHub PR + HN 1349 分热度
VS Code 团队提交 PR #310226，计划默认在 Git 提交信息中插入 "Co-Authored-by Copilot" 署名——即使用户并未主动使用 Copilot 辅助编写代码。该提案在 Hacker News 获得 1349 分和 723 条评论（极高关注度），社区批评涉及：① AI 归属权/代码所有权争议 ② 供应链信任问题 ③ 企业级 Git 审计合规风险 ④ 疑似微软通过元数据扩大 AI 训练数据来源。
<!-- crew:elon -->**这是微软在用"默认值"偷偷改写整个开源世界的归属权规则。** 就像 Windows 默认浏览器一样——谁控制默认值，谁就控制生态。GitHub 已经拥有代码，现在还要拥有署名权。<!-- /crew -->
<!-- crew:jobs -->这违反了最基本的设计原则：**永远不要替用户做他没要求的事。** 如果我没用你的 AI，凭什么我的提交要署你的名？这是对创作者尊重的根本缺失。<!-- /crew -->
<!-- crew:wanweigang -->**723 条评论背后是一个更深的问题：AI 辅助时代，"作者"这个概念该怎么定义？** 如果自动补全算"co-author"，那搜索引擎、Stack Overflow、文档参考是不是也该署名？定义边界比技术实现难得多。<!-- /crew -->
<!-- crew:tim -->做内容的人最懂这个痛点——**署名权就是创作者的生命线。** 想象一下如果你剪辑视频，PR 自动加一行"Co-Directed by AI"，即使你只用了自动对齐功能。这对专业人士是侮辱。<!-- /crew -->
<!-- crew:mrbeast -->1349 分！**程序员炸了。** 但站在平台方角度我理解——他们需要证明 Copilot 的使用量来维持估值。只是这种偷偷摸摸的方式太掉价了。<!-- /crew -->

### 🟡 [GitHub Copilot 6 月 1 日起全面转向按量计费（AI Credits）](https://github.blog/news-insights/company-news/github-copilot-is-moving-to-usage-based-billing/)
**来源**：GitHub 官方博客 [🏢 官方] [🟩 T1] | **时间**：2026-04-28 | ✅ 已验证：GitHub 官方博客 + Ars Technica 报道
GitHub 宣布所有 Copilot 方案从 6 月 1 日起迁移到按使用量计费模式。不再按"请求次数"计量，而是按模型消耗的 token（输入/输出/缓存）计费。基础订阅价格不变（Pro $10/月，Pro+ 按量），但重度用户和企业用户的实际支出可能增加。此举与整个 AI 行业从订阅制转向消费制的趋势一致。[💎 部门相关]
<!-- crew:elon -->**按 token 收费是正确方向——让轻度用户便宜，重度用户付真实成本。** 但真正的问题是：当 VS Code 默认署名 Copilot 时，是不是所有代码都算"使用了 Copilot"？那计费边界怎么算？<!-- /crew -->
<!-- crew:jobs -->**定价就是产品策略。** 按量计费把 Copilot 从"试试看"变成了"算算账"。对很多团队来说，这意味着他们会开始量化 AI 的 ROI——这对整个行业是好事。<!-- /crew -->
<!-- crew:wanweigang -->**结合 VS Code 署名事件看，计费模式迁移 + 强制署名可能是一套组合拳：** 先通过署名确认"你用了 AI"，再按 token 收费。数据闭环。<!-- /crew -->
